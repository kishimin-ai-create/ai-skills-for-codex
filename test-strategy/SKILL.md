# Test Strategy

テストの目的は成功を示すことではなく、重要な誤りを検出し、変更の安全性を説明すること。リスク、契約、実行コスト、証拠を一つの計画にする。

## 計画手順

1. 重要なユーザーフロー、データ損失・権限・金銭・可用性のリスク、受入条件を特定する。
2. 要件をUnit/Component/Integration/Contract/E2E/Manualへ割り当て、各層の責任と重複を表にする。
3. PRの高速ゲート（型、lint、Small、主要Integration）、マージ・リリースゲート（Medium/Large、a11y、性能、セキュリティ）を分ける。
4. fixtures、環境、秘密情報、再現性、並列性、flaky分類、失敗成果物（trace・log・screenshot）を決める。
5. 失敗時の停止条件、ロールバック、未検証項目、手動承認を明記する。
6. 変更後に実行結果と残存リスクを更新し、数値だけで品質を断定しない。

## 出力契約

リスクと受入条件、層別テスト一覧、実行順・品質ゲート、結果、未完了、残存リスク、次の判断を報告する。ミュータント分析などの専用品質分析は対象外とする。
