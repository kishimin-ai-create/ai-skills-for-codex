# Test Driven Development

## 目的

> このアプローチは、**twada** が推奨する TDD の実践スタイルに基づいている。
> TDD はテスト技法ではなく、**設計技法**である。小さなサイクルを回すことで、
> 「動作するきれいなコード」に到達することを目指す。

### すべてのコード変更に適用

TDD は、新機能の追加だけでなく、**すべてのコード変更に必須**とする。

| 変更の種類                    | TDD サイクル                                                                                          |
| ----------------------------- | ----------------------------------------------------------------------------------------------------- |
| 新機能                        | Red → Green → Refactor                                                                                |
| バグ修正                      | バグを再現する失敗するテストを書く → Green → Refactor                                                 |
| リファクタリング              | 最初にすべてのテストが成功することを確認してからリファクタリングする（作業中は常に Green を維持する） |
| レガシーコード                | 既存の振る舞いを特性テストで固定する → Green を確認する → TDD サイクルで変更する                      |
| パフォーマンス / セキュリティ | 問題を明らかにするテストを書いてから修正する                                                          |

> 変更を正当化する失敗するテストを先に用意せずに、プロダクションコードを変更してはならない。
> 失敗するテストを伴わない修正は TDD ではない。

## 手順

1. カバーしたいテストシナリオの一覧（テストリスト）を作成する。

2. テストリストから1つだけ選び、具体的で実行可能なテストに変換する。その後、そのテストが失敗することを確認する。
   - `test:` プレフィックスでコミットする。

3. 新しいテストと既存のすべてのテストが成功するようにプロダクションコードを変更する。新たに発見したシナリオがあれば、テストリストに追加する。
   - `feat:` プレフィックスでコミットする。

4. 設計を改善するため、必要に応じてコードをリファクタリングする。
   - `refactor:` プレフィックスでコミットする（複数回行ってもよい）。

5. テストリストが空になり、すべてのテストが成功し、対象となるカバレッジレポートの全体サマリーで報告されるすべての指標が 80% 以上になるまで、ステップ2から繰り返す。
