# Frontend Display

表示設計は、対象ブラウザ、利用者、ネットワーク、言語、テーマ、アクセシビリティを要件として固定してから実装する。特定の製品や画面だけの前提をSkillへ持ち込まない。

## 表示設計ワークフロー

### 1. 対象環境と要件を固定する

- サポートするブラウザ、OS、端末幅、入力方式、最低ネットワーク条件を確認する。
- デスクトップ、タブレット、モバイルのレイアウト境界と、ズーム・文字サイズ変更時の要件を決める。
- 対応言語、日付・数値・通貨・タイムゾーン、RTL、翻訳文の伸長を確認する。
- ライト／ダーク／OS連動テーマ、ユーザー設定の保存、初期表示時のちらつきを確認する。
- 画像・フォント・動画の品質、容量、著作権、代替テキスト、遅延読み込みを確認する。

### 2. レイアウトを設計する

- まずコンテンツの優先順位と最小幅を定め、固定幅ではなく流動的な幅・余白・グリッドを使う。
- ブレークポイントは端末名ではなく、コンテンツが破綻する幅で決める。
- 長い翻訳、拡大表示、キーボードフォーカス、タッチ操作でも情報が隠れないことを確認する。
- 表示専用コンポーネントとデータ・ビジネスロジックを分離する。

### 3. メディアとテーマを実装する

- 画像は用途に応じた形式・サイズ・レスポンシブ候補を選び、重要度に応じて優先読み込みと遅延読み込みを使い分ける。
- 内容を伝える画像には意味のある代替テキストを付け、装飾画像は支援技術から隠す。
- テーマ切替は色だけで意味を伝えず、コントラスト、フォーカス、境界、状態表示を両テーマで確認する。
- フォント読み込み、CLS、LCP、キャッシュ、圧縮、プリロードの効果を測定してから最適化する。

### 4. ローカライズを実装する

- 描画時に現在のロケールの翻訳関数を使い、特定言語のメッセージを直接フォールバックにしない。
- 日付・数値・通貨はロケール対応のフォーマッターを使い、文字列連結で組み立てない。
- 翻訳の長さ、複数形、性、タイムゾーン、RTL、未翻訳キーを検証する。
- document title、favicon、aria-label、alt、ローディング・エラー文言もUIとして翻訳する。

### 5. 検証して記録する

- 代表的なブラウザ・端末・幅・ズーム・言語・テーマ・ネットワークで表示を確認する。
- キーボード、スクリーンリーダー、タッチ、色覚、強制色、 reduced motion を確認する。
- Lighthouse等でCore Web Vitals、画像容量、フォント、キャッシュ、レイアウトシフトを測定する。
- 対応範囲、非対応範囲、例外、測定結果、残存リスクを設計記録へ残す。

## 安全性の基準

- viewportのズームを無効化しない。利用者が拡大しても操作可能なレイアウトを保つ。
- 色だけ、ホバーだけ、音だけで状態や意味を伝えない。
- 本番の機密情報、デバッグ用ソースマップ、不要な内部URLを公開しない。
- パフォーマンス最適化で可読性、代替テキスト、キーボード操作、翻訳を犠牲にしない。
- 対応ブラウザを根拠なく増減させず、実利用と保守コストで判断する。
