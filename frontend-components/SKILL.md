# Frontend Components

コンポーネントは、表示責務とビジネス責務を分離し、安定したインターフェースとCompositionで組み立てる。既存フレームワーク規約とプロジェクトのUI契約を先に確認する。

## コンポーネント設計ワークフロー

### 1. 責務と所有者を決める

- アプリ全体で共有するUI（Button、Input、Dialogなど）と、機能・ドメインが所有するUIを分類する。
- `ui/`や`shared/components/`には表示とアクセシビリティの責務だけを置き、ページロジック、ルーティング、グローバル状態、データ取得を直接依存させない。
- `features/`やページコンテナには、データ取得、状態管理、ドメイン変換、ルーティング連携を置く。
- 実際に繰り返し利用されることを確認するまで、ドメインUIを共通化しない。

### 2. Propsとイベント契約を設計する

- 共有UIは対応するネイティブHTML要素の標準props・イベントを拡張し、そのまま渡せる型にする。
- `onClick`、`disabled`、`type`、`aria-*`などを独自シグネチャで再定義しない。
- 意味のある名前、明確なデフォルト、制御／非制御の方針、キーボード操作、フォーカス、ラベルを定義する。
- 一つの巨大なGod Objectではなく、小さく単一責務の部品をCompositionで組み合わせる。

### 3. 状態と非同期境界を明示する

- `idle`、`loading`、`success`、`error`など排他的な状態はDiscriminated Unionで表現し、不正な組合せを型で排除する。
- 非同期処理は対象に近い`Suspense`境界で待機させ、必要に応じて`ErrorBoundary`を直上に置く。
- ページ全体を一つのローディング表示で覆わず、操作対象の粒度でフォールバックを設計する。
- `Suspense`非対応の取得処理と組み合わせない。仕様で定めた初回表示と部分更新のローディングを守る。

### 4. パターンとライブラリを選ぶ

- まず既存のコンポーネントライブラリを評価し、不足部分だけHybridで補う。
- Full scratch、Compound components、Render props、Factoryなどは、重複削減や責務分離という具体的効果がある場合だけ採用する。
- ライブラリのアクセシビリティ、バンドルサイズ、カスタマイズ性、ベンダー依存、保守体制を比較する。

### 5. 配置・命名・検証を行う

- フレームワーク標準の規約を優先し、共有UIと機能所有UIの境界をディレクトリに反映する。
- コンポーネントは`PascalCase`、props・イベント・変数は`camelCase`、CSSクラス・フォルダは`kebab-case`を基本とする。
- 正常、loading、error、空状態、キーボード、スクリーンリーダー、レスポンシブ、ロケールを検証する。
- APIやフィルターの契約（例：`page`、`pageSize`、`totalCount`、`onPageChange`）を一部だけ扱わず、関連値を常に一組で維持する。

## 安全性の基準

- 表示ロジックとビジネスロジックを混在させず、サービス、Hook、Moduleへ分離する。
- `aria-*`、ラベル、十分なタップ領域、明確なリンク名などアクセシビリティを後付けにしない。
- 多言語UIでは翻訳関数を通して表示し、ローディング文言、タイトル、favicon、代替テキストのロケールを揃える。
- 状態モデル、ページネーション、データ取得、エラー境界の変更を、無関係な機能追加や全面的なスタイル変更と混ぜない。

## 参照資料を選ぶ

- 既存の詳細なカテゴリ、Suspense、状態、命名、アンチパターン、ディレクトリ例を確認する場合は[component-pattern-catalog.md](./references/component-pattern-catalog.md)を読む。
- ネイティブHTML props互換性、`ui`・`features`・`pages`の責務境界を確認する場合は[component-contracts.md](./references/component-contracts.md)を読む。
- 実装・レビュー時は、参照資料の契約とプロジェクト固有のUI仕様を照合する。

## 出力契約

1. コンポーネントの責務、所有者、配置先
2. Props・イベント・状態・非同期境界の契約
3. 採用したパターン／ライブラリと採用しなかった代替案
4. アクセシビリティ、i18n、レスポンシブ、既存API契約への影響
5. テスト項目、残存リスク、ロールバック方針
