# Routing Decision Guide

## 方式選択

- SEOと初期表示が重要な公開ページはSSRまたはSSG/再検証を検討する。
- ログイン後の高度な操作が中心で検索対象でない画面はCSRを検討する。
- 認証・権限境界や秘密情報を扱う取得はサーバー側を優先する。
- 更新頻度、キャッシュ、データ鮮度、障害時の表示を比較して方式を決める。

## URLへ置く情報

- 置く: ページ番号、検索条件、並び順、公開リソース識別子、ロケール。
- 置かない: トークン、秘密情報、個人情報、短命な内部状態、認可判断の根拠。

## 認証遷移

1. 未認証なら、許可された相対return URLを保持してログインへ遷移する。
2. セッション期限切れは再認証へ遷移し、元の操作を安全に再開できるか定義する。
3. 認証済みでも権限不足なら403相当の表示とし、ログインへ無限遷移させない。
4. サーバー側で最終的な認可を行う。

## 採用を見送る条件

- UIの非表示だけで権限を保護する。
- 任意URLへリダイレクトする。
- 全ルートを同一の認証・キャッシュ・エラー処理で済ませる。
- ルート変更とAPI、データ移行、無関係なUI変更を一度に行う。
