# Frontend Routing

ルーティングはURL、認証境界、データ取得、表示状態、ブラウザ履歴を結ぶ公開契約である。フレームワークの既定に任せる部分と、アプリ固有の規則を分離して設計する。

## ルーティング設計ワークフロー

### 1. URL契約を定義する

- 公開ページ、認証が必要なページ、権限別ページ、API、404、エラー、リダイレクトを一覧化する。
- URLの命名、階層、動的セグメント、クエリ、末尾スラッシュ、大文字小文字、ロケールの規則を決める。
- URLがブックマーク・共有・再読み込みされても同じ状態を再現できるようにする。
- URLへ含めてよい情報と、機密情報・一時状態として含めてはならない情報を分ける。

### 2. 認証・認可境界を置く

- 未認証、認証済み、権限不足、セッション期限切れを別の遷移として定義する。
- サーバーまたはルート境界で認証を検証し、クライアント表示だけで保護したことにしない。
- リダイレクト先のオープンリダイレクト、認証情報漏えい、無限ループを防ぐ。
- 権限判定はサーバーの実データとポリシーを基準にし、UIの非表示だけを認可とみなさない。

### 3. ナビゲーションと履歴を設計する

- リンク、戻る、進む、直接URL、外部リンク、フォーム送信、未保存変更の挙動を確認する。
- 一時的なUI状態は履歴へ追加するか、置換するかを明示する。
- ローディング中の二重送信、競合する遷移、キャンセル、フォーカス移動を扱う。
- ページタイトル、見出し、フォーカス、スクリーンリーダー向け通知を遷移後に更新する。

### 4. データ取得とレンダリングを決める

- ルートごとにサーバー取得、クライアント取得、静的生成、再検証の理由を記録する。
- ルートパラメータ・クエリ・ロケール・認証情報をデータ取得へ安全に渡す。
- ローディング、空状態、404、権限不足、取得失敗、再試行をルート単位または機能単位で表示する。
- キャッシュ、再検証、Abort、古いデータ、重複リクエスト、ストリーミング境界を確認する。

### 5. SEOと配信を検証する

- 公開ルートのtitle、description、canonical、robots、sitemap、構造化データを定義する。
- 認証ページや一時URLを検索インデックスへ露出させない。
- 深いURLの直接アクセス、SPAフォールバック、末尾スラッシュ、言語切替、OGPを検証する。
- 代表的なブラウザ、キーボード、低速回線、エラー応答、履歴操作をテストする。

## 安全性の基準

- クライアント側のガードだけで認証・認可を完了させない。
- リダイレクト先を無検証で受け取らず、許可された相対パスまたは allowlist を使う。
- URL、クエリ、Referer、ログ、エラー画面へ秘密情報を出さない。
- ルーティング変更に、無関係なUI変更、API仕様変更、データ移行を混在させない。
