# Article Patterns

読者が目的の情報へ到達し、根拠と再現手順を確認できる記事を設計する。記事種別そのものを目的にせず、読者の課題と期待する行動から構成を決める。

## 執筆ワークフロー

### 1. 入力条件を確定する

次を執筆前に確認し、不明なものは仮定として記録する。

- 対象読者の知識レベルと前提
- 読者が解決したい課題または記事を読む目的
- 記事種別、媒体、公開形式、想定文字量
- 使用するリポジトリ、コード、ログ、仕様、一次資料
- 対象バージョン、実行環境、日付、地域、言語
- 事実確認、引用、スクリーンショット、機密情報の制約
- 読了後に読者へ期待する行動

### 2. 記事種別を選ぶ

[article-types.md](./references/article-types.md)から最も近い型を選ぶ。複数に該当する場合は主目的を1つ決め、補助的な型は節として組み込む。ユーザーが構成を指定した場合は、指定を優先し、欠落する前提だけ補う。

### 3. アウトラインを先に作る

タイトルと見出しだけで、次を確認できるアウトラインを作る。

- 読者の問いがどの節で解決されるか
- 問題、根拠、手順、結果、限界の順序が論理的か
- 事実、推測、意見、未検証事項が区別されているか
- 重要な結論が、前提や検証より先に断定されていないか
- 1節が1つの問いまたは作業を扱っているか

### 4. 根拠と再現性を整える

- 変化し得る情報は確認日、バージョン、環境を記録する。
- 外部資料は一次資料を優先し、URLと参照箇所を残す。
- コード、コマンド、設定、入力、期待結果を揃え、読者が追試できるようにする。
- 実行していないコードや確認していない結果を、実行済みとして書かない。
- 秘密情報、個人情報、内部URL、実在の認証情報を例へ含めない。
- 記事中の数値、日付、製品名、API名、画面名を本文全体で統一する。

### 5. 本文を書く

- 冒頭で対象読者、解決すること、解決しないことを短く示す。
- 最初に結論または完成状態を示し、その後に背景と手順を説明する。
- 見出しは内容を予測できる名詞または動詞で書く。
- 1段落1論点とし、長い説明は表、箇条書き、コードブロックへ分ける。
- 手順には前提、操作、期待結果、失敗時の確認を含める。
- コード例には言語識別子を付け、必要な省略はコメントまたは本文で明示する。
- 読者の環境差が結果へ影響する場合は、分岐と判断条件を書く。

### 6. 媒体に合わせて整形する

[writing-and-formatting.md](./references/writing-and-formatting.md)を参照し、Markdown、技術ブログ、開発ログ、リリースノートなどの制約を適用する。媒体の装飾を増やして、事実や手順の追跡可能性を損なわない。

### 7. 公開前にレビューする

[review-checklist.md](./references/review-checklist.md)を使い、内容、根拠、コード、リンク、表現、アクセシビリティ、機密情報、更新期限を確認する。レビューで残った未確認事項は、記事の限界として明記する。

## 品質原則

- 読者の課題に直接関係しない背景を増やさない。
- タイトル、見出し、要約、本文、まとめで約束を一致させる。
- 成功例だけでなく、失敗条件、制約、ロールバックまたは代替策を示す。
- 手順と理由を分け、コードはHow、検証はWhat、結論はWhyを伝える。
- 比較や推奨では、評価軸、対象範囲、トレードオフ、再考条件を明示する。
- 読者を不必要に煽らず、断定の強さを根拠の強さに合わせる。
- 翻訳・ローカライズでは、コード、識別子、コマンド、URL、プレースホルダーを変更しない。
- 画像や図だけに重要情報を置かず、本文または代替テキストでも伝える。

## 記事種別ごとの最低成果物

| 種別 | 最低限含める内容 |
| --- | --- |
| 製品・機能検証 | 対象環境、検証条件、観測結果、制約 |
| エラー解決 | 症状、原因、再現条件、修正、再発防止 |
| 技術課題の解決 | 背景、制約、選択肢、実装、検証、トレードオフ |
| 開発ログ | 目的、時系列、判断、変更、結果、未完了事項 |
| リリースノート | 変更、対象者への影響、移行・互換性、既知の問題 |
| 設計説明 | 問題、決定、代替案、理由、影響、再考条件 |

## 禁止事項

- 根拠のない「必ず」「完全」「最速」などの断定をしない。
- 実行していない手順、未確認の互換性、存在しないリンクを事実として書かない。
- 読者の環境にない秘密情報や個人情報を例へ埋め込まない。
- コードブロック内の識別子、コマンド、URL、プレースホルダーを翻訳・装飾目的で変更しない。
- 見出しレベルを見た目だけで飛ばさない。
- 長い引用で本文を置き換えない。必要な短い引用以外は要約し、出典を示す。
- 記事の種類に不要なテンプレート節を機械的に追加しない。

## 出力契約

記事と併せて、必要に応じて次を報告する。

- 選択した記事種別と選択理由
- 対象読者、前提、検証環境
- 参照した一次資料と確認日
- 実行した検証と結果
- 未検証事項、制約、更新が必要な情報
- 媒体固有の整形を適用した箇所
