# Fix Patterns

修正は症状を隠すことではなく、再現可能な根本原因を除去し、同じ契約違反を検出する回帰防止を追加すること。プロジェクト固有のランナー・規約・ADRを先に確認する。

## 手順

1. 依頼、エラーログ、再現手順、対象環境、変更範囲、受入条件を収集する。ログや会話にない事実を補完しない。
2. 最小の再現を作り、期待値と実測値、発生頻度、影響範囲、最初に壊れた境界を記録する。
3. 原因を実装、テスト、環境、依存、データ、仕様のいずれかに分類し、仮説ごとに検証する。最初のエラーと派生エラーを分離する。
4. 互換性・セキュリティ・性能・データ移行への影響を確認し、最小の変更案と採用しない代替案を比較する。
5. まず回帰テスト（失敗を表すテスト）を追加し、最小修正を実装する。テストを弱める、例外を握りつぶす、無条件retryやtimeoutで隠す修正は行わない。
6. 関連するUnit/Integration/E2E、型検査、Lint、buildを段階的に実行する。失敗は原因未確定を含めて報告する。
7. 差分、検証結果、残存リスク、ロールバック方法、未解決事項を記録する。

## 代表パターン

- 契約違反：status、error code、schema、nullabilityを境界で正規化し、成功HTTPでも不正bodyを成功扱いにしない。
- 状態競合：Abort、世代番号、idempotency key、トランザクションで古い応答や重複副作用を防ぐ。
- 入力・境界：空、最大長、Unicode、時刻・タイムゾーン、権限、ページ境界を回帰テストにする。
- 非同期：loading/empty/error/retry/cancelを明示し、実時間sleepや共有mutable fixtureに依存しない。
- UI回帰：利用者向けrole/label、フォーカス、キーボード、レスポンシブ状態を検証する。
- 依存・環境：lockfile、設定、OS/ブラウザ差、秘密情報を確認し、環境だけの失敗とコードの失敗を分ける。
