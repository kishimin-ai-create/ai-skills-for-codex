# Commit Message Rules

## 形式

```text
<type>: <summary>

<body — Why を説明する>
```

- Summary は命令形で、50文字以内にする（例: `add`、`fix`、`refactor`）。

## Type

- `test:` — テストを追加または更新する。
- `feat:` — テストを成功させるために新しい振る舞いを追加する。
- `refactor:` — 振る舞いを変更せずに構造を改善する。
- `fix:` — バグを修正する。
- `docs:` — ドキュメントのみを変更する。

## 本文（Why）

- 本文では**Why**を必ず説明する。
- **How**を説明しない。
- コードを見れば明らかなことを繰り返さない。
- Why が単に「仕様で要求されているから」になる場合は、代わりに**何が可能になったのか**を書く。
  - Bad: `仕様で要求されているため、顧客一覧機能を追加するため`
  - Good: `ユーザーが顧客一覧を確認できるようにするため`

## 言語

- すべてのコミットメッセージを英語で記述する。
- 明確で簡潔な英語を使用する。
- 曖昧または不明確な表現を避ける。

## コミットの粒度

- 1つのコミットは、1つの論理的な変更を表すものとする。
- コミットは小さく段階的に行う。
- リファクタリング、機能実装、バグ修正を同じコミットに混在させない。
- フォーマットのみの変更と機能的な変更を分離する。
- 可能な限り、ファイルの移動・名前変更とコードの変更を分離する。
- 自動生成されたコードは、意図的な変更と区別できるよう独立したコミットに分ける。
- レビューしやすく、元に戻しやすいコミットを維持する。
- レビュアーがコミットメッセージと差分だけで、そのコミットの意図を理解できるようにする。

小さすぎるコミットも避ける。

コミットメッセージがコードを見れば明らかな内容を言い換えているだけの場合（例: `create PostController`）、コミットの粒度が細かすぎる可能性が高い。

各コミットでは、アプリケーションが動作可能な状態（クラッシュしない状態）を維持する。依存するファイルが不足した状態のコミットを避ける。

## コミット順序

履歴が意図的な開発の流れとして読めるように、コミットを論理的な実装順序で並べる。

- 依存されるものを、それに依存するコードより先に実装する。
- 関連する変更を連続して配置する。
- 複雑な履歴になることを避けるため、開発中は WIP であっても頻繁にコミットする。
- LLM を使用してコードを生成する場合は、コミットを整理された状態に保つため、1回の生成範囲を限定する。
- PR を作成する前に、必要に応じて `git rebase -i` を使用してコミット順序の修正や squash / split を行う。

以下の状態を避ける。

- あるコミットが、そのコミットまたはそれ以前のコミットに存在しない関数を呼び出している。
- 同じ変更に関する変更の間に、無関係なコミットが挟まっている。
- 自動生成された変更と手書きの変更が、1つのコミットに混在している。

## 禁止事項

- `update`、`fix stuff` などの曖昧なメッセージ。
- 1つのコミットに複数の関心事を混在させること。
- Why を説明せず、What または How だけを記述すること。

## 検証

コミット前に以下を確認する。

- Whyが明確である。
- コミットの目的が1つである。
- コミットメッセージと差分だけで意図を理解できる。
- 複数の目的が存在する場合は、コミットを分割する。

## 良い例

```text
chore: remove obsolete security skill file

Since I no longer need the skill I wrote myself, I’m asking my copilot to write a security skill.
```

## 悪い例

```text
chore: remove obsolete security skill file
```
