# Engineering Principles

原則を暗記した標語として適用せず、目的、制約、変更コスト、失敗時の影響を比較して設計判断を行う。原則同士が衝突する場合は、優先順位とトレードオフを明示する。

## 適用ワークフロー

### 1. 成果と制約を明確にする

- 何を変更し、何を変更しないかを定義する。
- 機能、品質属性、期限、チーム、依存関係、運用、セキュリティ、互換性の制約を列挙する。
- 現在の挙動、契約、データ境界、失敗モードを観測してから判断する。
- 推測と確認済みの事実を分ける。

### 2. 変更の大きさを決める

- 最小の安全な変更で目的を満たせるか確認する。
- YAGNIと将来の変更可能性を比較し、今必要な拡張点だけを残す。
- 既存コードへ変更を加える場合は、公開API、データ、ログ、メトリクス、テスト、運用手順への影響を洗い出す。
- 大きな設計変更は、可逆な段階へ分割する。

### 3. 原則を選ぶ

[principles-catalog.md](./references/principles-catalog.md)から、問題に直接関係する原則を2〜5個選ぶ。各原則について以下を記録する。

- この原則が守る価値
- 今回の問題へ適用する具体的な箇所
- 適用しない場合のリスク
- 他の原則や制約との衝突
- 原則が有効であることを確認する検証

原則の名前だけを列挙して結論にしない。

### 4. 設計・実装する

- 公開契約と内部実装を分離し、変更の影響範囲を小さくする。
- 抽象化は、既知の変化点または複数の実利用がある場合に追加する。
- データ、エラー、時間、並行性、権限、リソース所有権を明示する。
- 失敗を隠さず、呼び出し側が回復・再試行・中止を判断できるエラーを返す。
- コメントはコードから読み取れないWhy not（採用しなかった理由、危険な前提、外部制約）だけを書く。

### 5. テストと観測で検証する

- 振る舞いの期待値をテストで表現し、実装詳細への過剰な依存を避ける。
- 正常系、境界値、失敗、再試行、並行実行、権限、互換性をリスクに応じて検証する。
- 性能改善は測定前後のデータ、負荷条件、回帰影響を記録する。
- ログ、メトリクス、トレースで運用時に判断できる状態を作る。
- 変更の安全性を、レビュー、静的解析、テスト、実行確認、ロールバック計画で確認する。

## 原則の使い分け

| 状況 | まず確認する原則 | 追加で確認すること |
| --- | --- | --- |
| 小さな機能追加 | KISS、YAGNI、Clarity | 将来の変更点、テスト範囲 |
| 重複した実装 | DRY、PIE、SLAP | 抽象化による結合と可読性 |
| 大きな依存関係 | SOLID、Law of Demeter、Least Astonishment | 公開契約、差し替え、テスト容易性 |
| 性能問題 | Amdahl、Gustafson、Pareto | 実測、ボトルネック、コスト |
| 分散システム | CAP、分散計算の誤謬、Postel | 障害、整合性、タイムアウト、再試行 |
| チームの遅延 | Brooks、Bus Factor、Parkinson | 作業分割、待ち時間、知識共有 |
| 長期保守 | Code Will Be Changed、Lehman's Laws、Technical Debt | 変更頻度、移行計画、撤退条件 |
| レビューの対立 | First Principles、Inversion、Confirmation Bias | 事実、反証、検証可能な基準 |

## 設計時の境界

- KISSは単純さを選ぶ指針であり、必要な複雑性を隠す理由にしない。
- DRYは意味が同じ知識を統合する指針であり、似て見える偶然の重複を急いで共有しない。
- SOLIDは目的に応じて使い、分割そのものを品質とみなさない。
- YAGNIは未要求の機能を作らない指針であり、可逆性や安全性に必要な設計を削らない。
- Postelの法則を入力の無制限な受け入れや、曖昧なエラー処理の根拠にしない。
- 法則・経験則・メンタルモデルは、仕様や測定結果の代わりにしない。
- 原則が法的、セキュリティ、プライバシー、契約上の要求と衝突する場合は、要求を優先して理由を記録する。

## レビューと報告

[review-checklist.md](./references/review-checklist.md)を使い、次を報告する。

- 問題と影響範囲
- 適用した原則と具体的な判断
- 却下した選択肢と理由
- 実装・テスト・観測の結果
- 残存リスク、可逆性、再考条件

コードはHow、テストはWhat、コミットメッセージはWhy、コメントはWhy notを表現する。コミットやリモート操作を行う場合は[git-rules](../git-rules/SKILL.md)を確認し、明示的な依頼なしにpushしない。
