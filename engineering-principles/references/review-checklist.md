# Engineering Principlesレビュー・チェックリスト

変更の規模とリスクに該当する項目を確認し、未確認の項目は理由を記録する。

## 目的と制約

- [ ] 変更する範囲と変更しない範囲が明確である。
- [ ] 要求、利用者、品質属性、期限、互換性、運用制約が記録されている。
- [ ] 観測した事実と仮定が分けられている。
- [ ] 現在の挙動、契約、失敗モードを確認している。

## 原則と判断

- [ ] 問題に直接関係する原則を選び、適用箇所を示している。
- [ ] 原則名の列挙だけでなく、守る価値と具体的な効果を説明している。
- [ ] 衝突する原則、制約、トレードオフを明示している。
- [ ] 却下した選択肢と理由がある。
- [ ] 判断が変わる再考条件がある。

## 設計とコード

- [ ] 変更は目的に対して最小で、不要な機能や抽象化を含まない。
- [ ] 意味の同じ知識だけを共有し、偶然の類似を過剰に統合していない。
- [ ] 責務、依存方向、公開契約、データ境界が明確である。
- [ ] 深い内部構造や暗黙の挙動へ過度に依存していない。
- [ ] 入力、出力、エラー、タイムアウト、再試行、冪等性を扱っている。
- [ ] 不正入力、権限不足、破壊操作、機密情報を安全側に扱っている。
- [ ] コメントは実装から読めないWhy notだけを説明している。

## テストと観測

- [ ] テストが期待する振る舞いを表し、実装詳細に過度に結合していない。
- [ ] 正常系、境界値、失敗、再試行、並行性、権限、互換性をリスクに応じて検証している。
- [ ] 変更前後の性能を同じ条件で測定している。
- [ ] ログ、メトリクス、トレースで障害を追跡できる。
- [ ] テスト結果、測定条件、未検証範囲を記録している。

## 変更と運用

- [ ] 公開API、データ、設定、依存、ログ、運用手順の影響を確認している。
- [ ] 変更を段階的に適用できる。
- [ ] ロールバックが定義され、データや契約をさらに壊さない。
- [ ] 移行期間、互換層、フラグ、終了条件が明確である。
- [ ] 知識や作業が一人に集中していない。
- [ ] 期限、作業分割、待ち時間、調整コストを見積もりへ反映している。

## 完了報告

- [ ] コードはHowを表現している。
- [ ] テストはWhatを表現している。
- [ ] コミットメッセージはWhyを表現している。
- [ ] 未解決の技術的負債と返済条件を記録している。
- [ ] 残存リスク、可逆性、次の検証を報告している。
