## 設計原則

| 原則                 | 判断に使う問い                                                         |
| -------------------- | ---------------------------------------------------------------------- |
| Code as Design       | コードから設計意図と境界を読み取れるか。                               |
| Code Will Be Changed | 将来の変更点を局所化し、変更を安全に試せるか。                         |
| KISS                 | 目的に必要な最小の複雑性になっているか。                               |
| DRY                  | 同じ知識を複数箇所で管理していないか。偶然の類似を誤統合していないか。 |
| YAGNI                | 要求されていない機能や抽象化を先に作っていないか。                     |
| PIE                  | 既存の仕組みを再利用して、重複を減らせるか。                           |
| SLAP                 | 同じ抽象レベルの処理を同じ場所へまとめているか。                       |
| SOLID                | 責務、拡張、置換、分離、依存の方向が変更に耐えられるか。               |

## アーキテクチャとシステムの法則

| 法則                               | 判断に使う問い                                                                     |
| ---------------------------------- | ---------------------------------------------------------------------------------- |
| Conway's Law                       | システム構造がチーム間の通信構造を反映していないか。望む構造と組織を整合させたか。 |
| Gall's Law                         | 複雑なシステムを、動く単純なシステムから段階的に進化させられるか。                 |
| Law of Leaky Abstractions          | 抽象化が隠しきれない下位層の制約を利用者へ適切に伝えているか。                     |
| Tesler's Law                       | 必要な複雑性をどの層が負担するかを意識しているか。                                 |
| CAP Theorem                        | 分断時に整合性、可用性、分断耐性のどれを選ぶか明示したか。                         |
| Fallacies of Distributed Computing | 遅延、失敗、帯域、重複、順序、管理権限の不確実性を仮定していないか。               |
| Postel's Law                       | 互換性と安全性を両立し、曖昧な入力を無制限に受け入れていないか。                   |
| Metcalfe's Law                     | 接続数の増加が価値と同時に調整コストや障害面を増やしていないか。                   |
| Zawinski's Law                     | 機能追加で既存の目的を損なうほど、システムを無制限に拡張していないか。             |
| Law of Unintended Consequences     | 直接の変更以外に生じる副作用と二次的影響を検証したか。                             |

## コード品質の法則

| 法則                            | 判断に使う問い                                               |
| ------------------------------- | ------------------------------------------------------------ |
| Hyrum's Law                     | 利用者が観測できる暗黙の挙動を契約として扱っているか。       |
| Boy Scout Rule                  | 触れた範囲を小さく改善し、目的外の大規模変更を避けているか。 |
| Broken Windows Theory           | 小さな品質劣化を放置して、低い基準を常態化していないか。     |
| Technical Debt                  | 意図的・偶発的な近道の返済条件、利息、期限を記録しているか。 |
| Law of Demeter                  | オブジェクト内部の深い構造へ過度に依存していないか。         |
| Principle of Least Astonishment | 利用者や保守者が予測する挙動と実際の挙動が一致するか。       |
| Kernighan's Law                 | デバッグしやすい単純な実装と観測可能性を確保しているか。     |
| Premature Optimization          | 測定前に複雑な最適化を導入していないか。                     |
| Interface Segregation Principle | 利用しないメソッドまで実装させる大きな契約になっていないか。 |
| Cunningham's Law                | 不確かな知識を検証可能な質問や実験へ変換しているか。         |

## チームと組織

| 法則・概念        | 判断に使う問い                                                   |
| ----------------- | ---------------------------------------------------------------- |
| Brooks's Law      | 遅れたプロジェクトへの人員追加が通信と教育の負担を増やさないか。 |
| Dunbar's Number   | 関係者数が増えたとき、調整単位と情報経路を分けているか。         |
| Bus Factor        | 重要な知識や作業が一人に集中していないか。                       |
| Ringelmann Effect | 人数増加で個人の協働効率が低下していないか。                     |
| Price's Law       | 作業や知識が少数の貢献者へ偏っていないか。                       |
| Putt's Law        | 技術的意思決定を、技術を理解しない管理構造だけに委ねていないか。 |
| Peter Principle   | 役割昇格後の責務に必要な能力と支援を確認したか。                 |
| Dilbert Principle | 成果の低い人を管理職へ移すだけの解決になっていないか。           |
| Linus's Law       | 十分な目と多様な利用者で検証できる状態を作っているか。           |

## 計画と見積もり

| 法則・概念             | 判断に使う問い                                             |
| ---------------------- | ---------------------------------------------------------- |
| Parkinson's Law        | 期限まで作業を膨張させず、完了条件を定義しているか。       |
| The Ninety-Ninety Rule | 終盤の統合、品質、例外処理を見積もりから落としていないか。 |
| Hofstadter's Law       | 自分の見積もりが楽観的になる不確実性を含めているか。       |
| Goodhart's Law         | 指標を目標にした結果、望ましくない最適化が起きないか。     |
| Gilb's Law             | 測定できる価値とフィードバックを段階的に得られるか。       |
| Second-System Effect   | 新しいシステムへ過剰な機能と複雑性を詰め込んでいないか。   |

## テスト

| 原則・現象        | 判断に使う問い                                                         |
| ----------------- | ---------------------------------------------------------------------- |
| Pesticide Paradox | 同じテストだけで安心せず、入力・環境・観点を変えて欠陥を探しているか。 |

| 法則            | 判断に使う問い                                             |
| --------------- | ---------------------------------------------------------- |
| Amdahl's Law    | 改善対象の並列化できない部分が全体の上限を決めていないか。 |
| Gustafson's Law | 問題規模を拡大したとき、並列化の価値を活かせるか。         |

## ソフトウェアの進化

| 法則           | 判断に使う問い                                                   |
| -------------- | ---------------------------------------------------------------- |
| Lehman's Laws  | 利用環境の変化に合わせて、進化・複雑性・品質を管理しているか。   |
| Sturgeon's Law | 多くが価値を持たない可能性を踏まえ、必要なものを選別しているか。 |
| Murphy's Law   | 起こり得る失敗を列挙し、検出・回復・影響低減を設計したか。       |

## 認知とメンタルモデル

| 概念                         | 判断に使う問い                                                       |
| ---------------------------- | -------------------------------------------------------------------- |
| Dunning-Kruger Effect        | 自分の理解不足を認識し、反証やレビューを求めているか。               |
| Hanlon's Razor               | 悪意と決めつける前に、誤り、制約、情報不足を確認したか。             |
| Occam's Razor                | 同じ説明力なら不要な仮定の少ない案を選んでいるか。                   |
| Sunk Cost Fallacy            | 過去の投資ではなく、今後の価値とコストで継続を判断しているか。       |
| The Map Is Not the Territory | モデル、図、仕様が実システムや利用者体験と異なる可能性を確認したか。 |
| Confirmation Bias            | 仮説を支持する証拠だけでなく、反証するテストを行ったか。             |
| Hype Cycle                   | 流行語や期待値と、検証済みの能力・成熟度を区別しているか。           |
| Amara's Law                  | 短期の効果を過大評価し、長期の影響を過小評価していないか。           |
| Lindy Effect                 | 長く使われた手法の継続性を参考にしつつ、現在の適合性も検証したか。   |
| First Principles Thinking    | 慣習ではなく、目的、制約、観測可能な事実から分解したか。             |
| Inversion                    | 成功方法だけでなく、失敗・破壊・回復不能になる条件から考えたか。     |
| Pareto Principle             | 主要な価値や問題の集中箇所へ、検証と改善の資源を配分したか。         |

## ソフトウェア品質属性

| 属性             | 確認する観点                                                   |
| ---------------- | -------------------------------------------------------------- |
| Changeability    | 変更箇所、依存、移行、ロールバックが局所化されているか。       |
| Interoperability | 境界の契約、形式、バージョン、エラーが他システムと整合するか。 |
| Efficiency       | 時間、メモリ、帯域、コストを目的に対して効率よく使うか。       |
| Reliability      | 期待する条件で一貫して動き、失敗から回復できるか。             |
| Testability      | 重要な振る舞いを分離・観測・再現して検証できるか。             |
| Reusability      | 文脈を壊さず、意味のある契約として再利用できるか。             |

## コード構造

| 原則                  | 判断に使う問い                                               |
| --------------------- | ------------------------------------------------------------ |
| Simplicity Principle  | 理解と変更に必要な構造だけを残しているか。                   |
| Isomorphism Principle | 同じ問題構造をコード、データ、テストで一貫して表現できるか。 |
| Symmetry Principle    | 対になる操作や概念が予測可能な形で対応しているか。           |
| Hierarchy Principle   | 抽象度と責務の階層が明確で、境界を越えすぎていないか。       |
| Linearity Principle   | 制御とデータの流れを追跡でき、不要な分岐を避けているか。     |
| Clarity Principle     | 名前、構造、エラーが意図を最小の推測で伝えるか。             |
| Safety Principle      | 不正入力、失敗、権限、破壊操作を安全側へ扱うか。             |
