# Security: Configuration and Data Protection Rules

## Secret Management (V13.3)

- パスワード、APIキー、トークン、鍵素材などの秘密情報は、すべて鍵保管庫（Secret管理）に保存する。ソースコードやビルド成果物へ埋め込まない。
- 秘密情報へのアクセスは最小権限とし、アクセスを監査する。
- 文書化したスケジュールで秘密情報をローテーションし、自動期限切れを設定する。
- L3では鍵素材の保存と暗号処理にHSMを使用する。

## Backend Communication (V13.2)

- API、データベース、キューなどのバックエンド間通信は、個別サービスアカウント、短期トークン、またはmTLSで認証する。共有認証情報や静的APIキーを使わない。
- バックエンドのサービスアカウントには最小権限を付与する。
- `root/root`、`admin/admin`などの既定認証情報を使わない。
- アプリケーションが接続できる外部ホスト、パス、ポートの送信先許可リストを定義・強制する。

## Hardening and Information Leakage (V13.4)

- Webルートへ`.git`、`.svn`などのソース管理メタデータを配置しない。
- 本番の全コンポーネントでデバッグモードを無効にする。
- 本番でHTTP `TRACE`を無効にする。
- ディレクトリ一覧を公開しない。
- 意図していない内部APIドキュメントや監視エンドポイントを公開しない。
- バックエンドコンポーネントのバージョン番号をレスポンスに露出しない。

## Data Classification (V14.1)

- すべての機密データを特定し、保護レベルで分類する。
- 各レベルについて、暗号化、完全性、保持期間、ログ方針、アクセス制御の要件を文書化する。

## General Data Protection (V14.2)

- APIキーやセッショントークンなどの機密データをURLクエリへ入れず、本文またはヘッダーで送る。
- プロキシやロードバランサーによる機密レスポンスのキャッシュを防ぎ、使用後にパージする。
- ユーザー同意なしに機密データを第三者トラッカーや分析サービスへ送らない。
- 必要最小限のデータだけを返し、明示的な要求がない限り完全な値をマスクする（カード番号は末尾4桁のみ等）。
- 保持ポリシーを適用し、期限切れまたは不要な機密データを自動削除する。

## Client-Side Data Protection (V14.3)

- セッション終了時に、ブラウザDOMや`localStorage`などのクライアント保存領域から認証済みデータを消去する。可能なら`Clear-Site-Data`ヘッダーを使用する。
- 機密データを含むレスポンスへ`Cache-Control: no-store`を設定する。
- 機密データを`localStorage`、`sessionStorage`、`IndexedDB`、非セッションCookieへ保存しない。

## Review Evidence

- Secret管理、ローテーション、監査、送信先許可リストの設定を確認する。
- 本番ハードニングと機密レスポンスのキャッシュ防止を設定・テストで確認する。
- データ分類、保持期限、マスキング、クライアント消去をテストまたは設定証跡で示す。
