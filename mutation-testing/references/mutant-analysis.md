# Mutant Analysis

## 目的

生存したすべてのミュータントを個別に分析し、生存した原因を明らかにして適切に分類する。

推測ではなく、コードとテストから確認できる事実に基づいて判断する。

分類不能なミュータントを残してはならない。

## 手順

1. 変更前のコードを確認する。
2. ミュータントによる変更内容を確認する。
3. 変更前と変更後の意味的な差異を説明する。
4. 差異が発生する入力条件を特定する。
5. 現在のテストが差異を検出できなかった理由を確認する。
6. 次のいずれかに分類する。
   - Test Gap
   - Equivalent Mutant
   - Product Code Smell
   - Product Bug
   - Tool Limitation
   - Flaky / Timeout
7. 分類に応じた対応を行う。

## 分類

### Test Gap

不足している振る舞いを特定する。

その振る舞いだけを検証する最小のテストを追加する。

テストは仕様を表現するものとし、ミュータントだけを殺す不自然なテストを書いてはならない。

### Equivalent Mutant

変更前後の振る舞いがすべての有効入力で外部から観測可能な結果として等しいことを論理的に説明する。

根拠を記録する。

### Product Code Smell

テスト不足ではなく、プロダクトコードの設計や構造に原因がある場合に分類する。

コードは変更せず、問題を報告する。

### Product Bug

プロダクトコードの不具合を発見した場合に分類する。

テストを不具合に合わせて変更してはならない。

コードは変更せず、作業を停止して報告する。

### Tool Limitation

ミューテーションツールの制限によるものと判断できる場合に分類する。

再現可能な証拠を記録する。

### Flaky / Timeout

テストまたは実行環境に起因する問題を調査し、原因を記録する。

## 完了条件

- すべての生存ミュータントを分析している。
- すべての生存ミュータントを分類している。
- 分類の根拠を記録している。
- 分類不能なミュータントが存在しない。
