# Prohibited Actions

## 終了判断

以下を理由に作業を終了してはならない。

- Mutation Score が十分高い
- Raw Mutation Score が目標値を超えた
- カバレッジが高い
- 対象コードが重要ではない
- テストを書くことが難しい
- 生存ミュータントが少ない
- 問題なさそうに見える

すべてのミュータントを個別に確認する。

## プロダクトコード

ミュータントを殺すためにプロダクトコードを変更してはならない。

不具合やコード上の問題を発見した場合は、Product Bug または Product Code Smell として報告する。

テストを現在の不具合に合わせて変更してはならない。

## テスト

以下を行ってはならない。

- Assertion を弱くする
- テストを削除する
- テストを無効化する
- テストをスキップする
- TODO や未完成のテストを残す
- ミュータント専用の不自然なテストを書く
- 実装詳細だけを検証する
- 仕様と無関係なテストケースを追加する
- プロダクトコードの振る舞いをテスト内で再実装する

## Snapshot

以下を行ってはならない。

- Snapshot を無条件に更新する
- 差分を確認せずに Snapshot を承認する
- 期待仕様の変更を説明できない Snapshot 更新を行う
- Snapshot Test だけでミュータントを殺そうとする

## Mock

過剰な Mock によってプロダクトコードの振る舞いを再実装してはならない。

呼び出し自体が仕様である場合を除き、以下だけを確認してはならない。

```text
mock called
mock called once
mock called with
```

外部から観測可能な結果を検証する。

## Mutation Configuration

Mutation Score を上げるために以下を行ってはならない。

- mutation 対象を減らす
- mutator を無効化する
- exclude を追加する
- 対象ファイルを恣意的に除外する
- Not Covered をレポートから除外する
- Timeout を無視する
- 無効な設定でミュータントを生成しないようにする

対象範囲を狭める必要がある場合は、人間へ判断材料を提示する。

## ミュータント分析

以下を行ってはならない。

- 生存ミュータントを一括して問題なしと判断する
- 根拠なく Equivalent Mutant と宣言する
- 「見た感じ同じ」と判断する
- 「たぶん影響しない」と判断する
- 分類不能な状態で放置する
- 原因を調査せず Tool Limitation とする
- TIMEOUT を成功として扱う
- TEST ERROR を無視する

## AIによる変更

AIは Mutation Report を読む前にテストを追加してはならない。

次の順序を省略してはならない。

```text
Report
↓
Mutant
↓
Production Code
↓
Existing Test
↓
Missing Behavior
↓
Test Case
```

一般論だけを根拠にテストを追加してはならない。

追加した各テストと対象ミュータントの関係を説明する。

## 実行確認

以下を行ってはならない。

- 通常テストを実行せずに完了を報告する
- Mutation Test を再実行せずに完了を報告する
- 失敗したテストを残したまま完了を報告する
- 未調査のミュータントを残したまま完了を報告する
- 実行結果を確認せずに成功と報告する
