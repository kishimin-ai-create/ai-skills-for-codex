# Security: Cryptography Rules

## Algorithm Selection (V11.3)

- AES-GCMまたはChaCha20-Poly1305（認証付き暗号化）を使用し、ECBモードを使わない。
- 暗号化にPKCS#1 v1.5パディングを使わない。
- AEADまたはEncrypt-then-MACで暗号化データの完全性を必ず保護する。
- 同一鍵でnonceやIVを再利用しない。アルゴリズムに適した生成方法を使う。

## Hashing (V11.4)

- 承認済みの一般ハッシュはSHA-256、SHA-384、SHA-512、SHA-3とする。セキュリティ用途でMD5やSHA-1を使わない。
- パスワードはbcrypt、scrypt、Argon2idなど、計算コストのあるKDFで保存し、現在の指針に合わせてパラメーターを調整する。
- デジタル署名のハッシュは衝突耐性を持ち、出力長を256ビット以上にする。

## Random Values (V11.5)

- セッショントークン、秘密情報、nonce、OTPなどのセキュリティ用途の乱数にはCSPRNGを使い、`Math.random()`などを使わない。
- 生成値は128ビット以上のエントロピーを持たせる。
- UUID v4だけでは暗号学的秘密に十分ではない。単独では128ビットのセキュリティを保証しない。

## Key Management (V11.1)

- NIST SP 800-57に整合する鍵ライフサイクル方針を文書化し、適用する。
- すべての鍵、アルゴリズム、証明書と、使用可能・使用不可の範囲を暗号資産台帳で管理する。
- 秘密鍵を複数の主体で共有しない。共有秘密は最大2者に制限する。
- 耐量子暗号（PQC）アルゴリズムへの移行計画を定義する。

## Crypto Agility (V11.2)

- アルゴリズム、鍵長、暗号方式をコード全体の書き換えなしに交換できる設計にする。
- データの再暗号化と、任意時点での鍵ローテーションを実行できるようにする。

## Implementation Requirements (V11.2)

- 暗号プリミティブをゼロから実装せず、業界で検証された暗号ライブラリを使う。
- すべての鍵は128ビット以上のセキュリティを提供する（例: AES-256、RSA-3072、ECC-256）。
- サイドチャネルのタイミング攻撃を防ぐため、暗号処理を定時間で実行する。
- 暗号モジュールは安全側に失敗し、パディングオラクルなどのサイドチャネル情報をエラーから漏らさない。

## Key Storage

- すべての秘密情報と鍵素材を鍵保管庫またはHSMへ保存し、ソースコードやビルド成果物へ置かない。
- 機密鍵素材を扱う暗号処理には分離されたセキュリティモジュールを使用する。L3ではハードウェア保護されたHSMを使用する。

## Review Evidence

- 暗号資産台帳、鍵ライフサイクル、ローテーション、PQC移行計画を確認する。
- nonce/IV再利用、弱いアルゴリズム、CSPRNG未使用、自作実装がないことを確認する。
- 鍵保管場所、定時間比較、安全なエラー、改ざん・失敗時のテスト証跡を確認する。
