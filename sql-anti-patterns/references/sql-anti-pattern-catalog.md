# SQL Anti-pattern Catalog

## Logical design

- **Jaywalking**: 複数値をカンマ区切りや文字列へ埋め込む。交差テーブルや正規化した子テーブルを使用する。
- **Naive Trees**: 階層を親IDだけで表し、深い走査・移動・削除を複雑にする。再帰CTE、閉包テーブル、経路列など要件に合うモデルを選ぶ。
- **ID Required**: すべてのテーブルに意味のないIDを必須化する。自然キー、複合キー、業務上の一意制約を検討する。
- **Keyless Entry**: 主キーや一意制約がなく重複や特定不能な行を許す。行の識別と重複条件を制約化する。
- **EAV (Entity-Attribute-Value)**: 型・制約・検索性を失う属性汎用化。安定した属性は列へ、可変属性は型付きJSON等を制約付きで使用する。
- **Polymorphic Associations**: 複数テーブルを文字列型とIDで参照し、外部キーを張れない。交差テーブルまたは型ごとの外部キーを使う。
- **Multi-Column Attribute**: 値の個数ごとに列を増やす。行の繰り返しを子テーブルへ移す。
- **Metadata Tribbles**: 同じ意味のメタデータ表や設定表を増殖させる。単一の明確なメタデータモデルと制約を設計する。

## Physical design

- **Rounding Error**: 金額を浮動小数点で保存する。最小通貨単位の整数または固定精度Decimalを使う。
- **Thirty-One Flavors**: 同じ意味のデータ型・表記・単位が列ごとに乱立する。ドメイン型、CHECK、規約を統一する。
- **Phantom File**: DB外のファイルをデータの正本として参照し、バックアップ・整合性・権限を失う。DBまたは管理されたオブジェクトストレージと参照を設計する。
- **Index Shotgun**: 根拠なく大量のインデックスを作る。実行計画、選択性、更新コストを確認し、複合インデックスの順序を設計する。

## Queries

- **Fear of the Unknown**: NULLや未知値の三値論理を理解せず比較する。`IS NULL`、NULLの意味、デフォルトを明示する。
- **Ambiguous Groups**: GROUP BYと非集約列の意味が曖昧。全非集約列をグループ化するか、集約・ウィンドウ関数で意図を明示する。
- **Random Selection**: 大量表で全行をランダムソートする。対象を絞り、DBMSに合うサンプリングやキー方式を使う。
- **Poor Man's Search Engine**: `%term%`やLIKE連結だけで全文検索を実装する。全文検索・適切なインデックス・検索サービスを要件に応じて選ぶ。
- **Spaghetti Query**: 1本の巨大クエリへ責務を詰め込む。CTE、ビュー、段階的なクエリ、アプリケーション境界で責務を分ける。
- **Implicit Columns**: `SELECT *`や暗黙列へ依存する。必要な列を明示し、API・ビューの契約を安定させる。

## Application development

- **Readable Passwords**: パスワードや秘密情報を平文・可逆形式で保存する。レビュー済みの適応的ハッシュと秘密情報管理を使う。
- **SQL Injection**: SQL文字列へ入力を連結する。パラメータ化クエリ、許可リスト、最小権限を使う。
- **Pseudo-Key Neat-Freak**: 外部キーや自然な関連を無視して人工キーだけで整合性を担保する。外部キーと業務上一意な制約を併用する。
- **See No Evil**: エラーを握りつぶし、失敗や整合性違反を隠す。安全な公開エラー、内部ログ、監視、ロールバックを設計する。
- **Diplomatic Immunity**: すべてのDBエラーを無視・汎用化し、原因を追跡できない。安全な公開情報と診断可能な内部情報を分ける。
- **Standard Operating Procedure**: DB操作を手順書だけで運用し、再現性・レビュー・ロールバックがない。バージョン管理されたマイグレーションと自動検証を使う。

## Mini anti-patterns

- CSV列を複数行へ分割する前提で保存する。
- 「自分の環境では動く」を移植性・照合順序・設定差の検証なしに受け入れる。
- BIGINTが常に十分だと仮定する。最大値、符号、生成戦略を確認する。
- 価格を浮動小数点で保存する。
- 予約語をテーブル・列名に使う。
- すべての列へインデックスを作成する。
- `NOT IN`とNULLの三値論理を考慮しない。
- DBMS固有SQLを移植可能と称する。
- 大量行を全件ランダム取得する。
- ハッシュ文字列の長さ・形式を考慮せずVARCHARへ保存する。
- クエリパラメータを引用符内へ埋め込む。
- グループごとの採番を競合安全でない方法で実装する。
- 構文エラーメッセージを読まずに修正を繰り返す。
- テーブル・列のrenameを依存関係・段階移行なしに行う。
- DBMS固有のStored Procedureを要件・運用性の検証なしに採用する。

## Foreign-key mini anti-patterns

### Standard SQL

- 参照方向を逆にする。
- 存在しないテーブルを参照する。
- 親側の一意でない列を参照する。
- 複合外部キーを個別制約へ分解する。
- 複合外部キーの列順を誤る。
- 外部キー列のデータ型を不一致にする。
- 文字照合順序を不一致にする。
- 孤児行を挿入する。
- NOT NULL列へ`SET NULL`を指定する。
- 制約識別子を重複させる。
- 互換性のないテーブル種別を混在させる。

### MySQL 8.0

- 互換性のないストレージエンジンを使う。
- 外部キー列へ過大なデータ型を使う。
- 一意でないインデックスを外部キー参照に使う。
- インライン参照構文のDBMS差を確認しない。
- デフォルト参照アクションを要件なしに使う。
- 互換性のないテーブル種別を混在させる。

## Review checklist

- 対象DBMSとバージョンを確認したか。
- 実行計画、統計情報、データ量、書き込み負荷を根拠にしたか。
- 入力がパラメータ化されているか。
- 制約、トランザクション、ロック、リトライ、冪等性を確認したか。
- マイグレーションが既存データ、オンライン実行、ロールバックを考慮しているか。
- 改善前後の結果、性能、整合性、権限、監視を検証したか。
