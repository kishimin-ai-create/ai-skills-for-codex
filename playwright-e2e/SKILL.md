# Playwright E2E Test

## 目的

Playwright を使用して、認証と主要リソースの重要なユーザーフローを実際の操作に近い形で検証すること。

## 手順

### 1. 必須シナリオを確認する

すべての feature ブランチには、以下のシナリオを検証する成功する Playwright テストを必ず含める。

これらは必須要件であり、削除してはならない。

#### 認証 — サインアップの正常系

- サインアップページに移動する。
- 必要な情報を入力する。
- フォームを送信する。
- アプリ一覧ページ、または同等の認証済み画面にリダイレクトされることを確認する。
- エラーメッセージが表示されていないことを確認する。

#### 認証 — サインインの正常系

- サインイン（ログイン）ページに移動する。
- 有効な認証情報（既存ユーザー）を入力する。
- フォームを送信する。
- アプリ一覧ページにリダイレクトされることを確認する。
- エラーメッセージが表示されていないことを確認する。

#### CRUD — 正常系

主要リソースについて、作成 → 参照 → 更新 → 削除のライフサイクル全体を検証する。

| ステップ | 操作                                         | 検証                                               |
| -------- | -------------------------------------------- | -------------------------------------------------- |
| Create   | 必要な情報を入力して新しいリソースを作成する | 作成したリソースが一覧または適切な画面に表示される |
| Read     | 作成したリソースを参照する                   | 作成した内容が正しく表示される                     |
| Update   | 作成したリソースを編集して保存する           | 更新した内容が正しく反映される                     |
| Delete   | 作成したリソースを削除する                   | 削除したリソースが表示されなくなる                 |

### 2. テストを構造化する

各テスト内では **AAA** 構造を使用する。

テスト名では、以下を表現する。

- 何をテストするのか
- どのような条件で
- どのような結果を期待するのか

```ts id="xk6nxm"
// Good
test('有効な認証情報をサインインページで送信した場合、ユーザーはアプリ一覧にリダイレクトされる', async ({ page }) => { ... })

// Bad
test('ログインできる', async ({ page }) => { ... })
```

### 3. 重複する UI 操作を抽出する

ページ移動、フォーム入力、ボタンクリックなど、繰り返し行う UI 操作をヘルパー関数または Page Object Model に抽出する。

ヘルパーはテストファイルの近く（`e2e/helpers/` またはテストと同じ場所）に配置する。

### 4. ユーザーから見える結果を検証する

`page.waitForSelector` よりも `expect(locator).toBeVisible()` および `expect(locator).toHaveText()` を優先する。

内部状態や DOM 構造ではなく、ユーザーから見える結果を必ず検証する。

### 5. セマンティックなセレクターを使用する

`locator('css')` や `locator('[data-testid]')` よりも、以下を優先する。

- `getByRole`
- `getByLabel`
- `getByPlaceholder`
- `getByText`

`data-testid` は最後の手段としてのみ使用する。

`data-testid` を使用する場合はコメントを追加する。

### 6. 実行環境を設定する

CI では、テストをローカル開発サーバーに対して実行する。

Playwright の設定では、必要に応じてステージング環境や本番環境を指定できるよう、環境変数から `baseURL` を設定する。

テストファイル内に `localhost:` をハードコードしてはならない。
