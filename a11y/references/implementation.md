## 適用基準

明示された要件がなければWCAG 2.2 Level AAを基準にする。WCAG 2.2は以前のWCAG 2.xとの後方互換を意図しているが、対象契約や法令が特定版を指定する場合はその版も個別に確認する。

代表的なAA基準を次のように扱う。

| 対象 | 目安 |
| --- | --- |
| 通常テキストのコントラスト | 4.5:1以上 |
| 大きなテキストのコントラスト | 3:1以上 |
| UI部品・意味のあるグラフィック | 隣接色に対して3:1以上 |
| ターゲットサイズ | 原則24×24 CSS px以上、または許容される間隔・例外を満たす |
| リフロー | 320 CSS px幅または256 CSS px高さ相当で、二次元配置が本質的な部分を除き情報・機能を失わない |

数値だけで判断せず、該当する成功基準の定義と例外を確認する。

## セマンティクスとARIA

- 操作にはネイティブコントロールを使用する。
- リンクは移動、ボタンは動作に使用する。
- 見出しレベルを見た目ではなく文書構造で決める。
- ページに主領域を1つ設け、必要に応じて`header`、`nav`、`main`、`aside`、`footer`を使う。
- 同種のランドマークが複数ある場合は、それぞれ識別できる名前を与える。
- ARIAロールを付ける前に、同じ暗黙ロールを持つHTML要素を使用できないか確認する。
- ARIAの状態やプロパティをUI状態と同時に更新する。
- ARIAを付けてもキーボード操作やフォーカス管理は追加されないため、必要な動作を別途実装する。

## 名前と説明

- 可視ラベルをアクセシブルネームの基礎にする。
- アイコンのみのボタンには、動作を表す名前を与える。
- 同じ名前の操作が複数ある場合は、行や対象を含めて区別可能にする。
- `aria-label`で可視ラベルと異なる名前を付けることを避ける。
- 長い補足説明は`aria-describedby`などで関連付け、名前へ詰め込まない。
- `title`属性だけに重要な名前や説明を依存させない。
- 名前計算の結果をブラウザーのアクセシビリティツリーで確認する。

## キーボードとフォーカス

- `Tab`と`Shift+Tab`で主要な操作へ論理順に移動できるようにする。
- `Enter`と`Space`の動作はネイティブ要素または該当ウィジェットの慣例に合わせる。
- 複合ウィジェットでは、通常は`Tab`でウィジェットへ入り、矢印キーで内部移動し、`Tab`で抜ける。
- `Escape`で閉じる操作を提供する場合は、閉じた後のフォーカスを起点へ戻す。
- 要素を削除した後は、近接する意味のある操作または結果見出しへフォーカスを移す。
- SPAの画面遷移では、画面タイトルを更新し、必要な場合は新しい主見出しまたは主領域へフォーカスを移す。
- スキップリンクを用意し、キーボードフォーカス時に視認可能にする。
- スクロール追従ヘッダーやオーバーレイでフォーカス表示を隠さない。

## フォームとエラー

- 各入力へ永続的な可視ラベルを関連付ける。
- 関連する入力群には`fieldset`と`legend`を使用する。
- 必須、入力形式、単位、制約を入力前に理解できるようにする。
- `autocomplete`へ適切なトークンを設定し、入力目的を機械可読にする。
- エラー時は入力値を保持する。
- エラーをフィールドと関連付け、必要に応じて`aria-invalid="true"`を設定する。
- エラー要約をページ上部に置く場合は、該当フィールドへのリンクを提供する。
- 送信後にエラー要約へフォーカスを移す場合は、再送信や修正への経路を明確にする。
- 成功メッセージは見える状態にし、フォーカスを移さない場合は適切に通知する。

## 動的コンテンツ

- 視覚的な更新が支援技術へ自動通知されるか確認する。
- 通常の進捗・成功通知には控えめなライブリージョンを使用する。
- 即時対応が必要なエラーだけを`role="alert"`などの割り込み通知にする。
- ライブリージョンは更新前からDOMに存在させる。
- 頻繁な更新をまとめ、読み上げを連続発生させない。
- ローディング中であることと、完了または失敗したことの両方を伝える。
- `aria-busy`を使用する場合は処理完了時に確実に解除する。

## 画像・アイコン・メディア

- 情報を伝える画像の代替テキストは、見た目の説明ではなく文脈上の目的を表す。
- 装飾画像は空の`alt`または適切な方法でアクセシビリティツリーから除外する。
- リンク画像やボタン画像の代替テキストは動作を表す。
- 複雑なグラフには短い代替テキストと、同等のデータまたは詳細説明を提供する。
- SVGを操作対象にする場合は、操作要素側へ確実な名前を付ける。
- 収録済み動画へ同期字幕を提供する。
- 音声のみの内容にはトランスクリプトを提供する。
- 自動再生を避け、再生・停止・音量を利用者が制御できるようにする。

## 色・コントラスト・表示

- 通常テキストは4.5:1、大きなテキストは3:1以上のコントラストを確保する。
- 入力境界、アイコン、選択状態、フォーカス表示など、理解に必要な非テキスト要素は3:1を基準に確認する。
- エラー、選択、必須、成功を色だけで示さない。
- ブラウザーズーム400%相当と狭いビューポートで横方向の読み取りを強制しない。ただし地図、データ表、図など二次元配置が本質的なものは例外を確認する。
- 利用者が行高、段落間隔、文字間隔、単語間隔を広げても内容を切り取らない。
- `forced-colors`で境界、状態、フォーカスが消えないようにする。
- `prefers-reduced-motion: reduce`では、目的に不要な移動・拡大・視差効果を停止または単純化する。
- 点滅や閃光を避ける。

## 複雑なウィジェット

ネイティブHTMLで要件を満たせない場合だけ、該当するARIA APGパターンを選ぶ。

実装前に以下を定義する。

- ロール、状態、プロパティ
- アクセシブルネームと説明
- `Tab`での出入り
- 矢印キー、`Home`、`End`、`Enter`、`Space`、`Escape`の動作
- 初期フォーカスと終了後のフォーカス
- 無効項目の発見可能性
- 動的追加・削除・選択変更の通知

APGのサンプルコードをそのまま本番利用せず、対象ブラウザーと支援技術で検証する。
