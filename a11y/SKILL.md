# Web Accessibility

Webインターフェースを、障害の有無や入力・出力手段にかかわらず利用できるように実装・評価する。

## 基準を決める

1. リポジトリの要件、対象市場、法的要件、既存ADRを確認する。
2. 明示された基準がなければ、WCAG 2.2 Level AAを既定の評価基準にする。
3. 適合レベル、対象画面、対象ブラウザー、支援技術、既知の制約を作業記録に残す。
4. APGの例を参考実装として扱い、規範要件はWCAG、WAI-ARIA、HTML仕様で判断する。

正式な適合認証や法的判断を求められた場合は、検査範囲と限界を明示し、資格を持つ監査担当者による確認を推奨する。

## 実装ワークフロー

### 1. ユーザーフローを特定する

- 主要操作、エラー経路、空状態、読み込み状態、権限不足状態を列挙する。
- マウス、キーボード、タッチ、音声入力、スクリーンリーダー、拡大表示で同じ目的を達成できるか確認する。
- コンポーネント単体ではなく、開始地点から完了状態までのフォーカスと通知を追跡する。

### 2. ネイティブHTMLを優先する

- `button`、`a`、`input`、`select`、`textarea`、`fieldset`、`legend`、見出し、ランドマークを本来の用途で使用する。
- `div`や`span`へクリック処理を付けてネイティブコントロールを再実装しない。
- ARIAは不足するセマンティクスだけを補うために使用する。ARIAによって要素の動作、フォーカス、キーボード操作が自動実装されると仮定しない。
- ロールと状態が実際の見た目・操作・現在状態と一致するようにする。

### 3. 名前・説明・構造を提供する

- すべてのインタラクティブ要素に、目的を識別できるアクセシブルネームを与える。
- 可視ラベルがある場合は、原則としてその文言をアクセシブルネームに含める。
- ラベル、補足説明、入力エラーを適切に関連付ける。
- 見出し階層、ランドマーク、リスト、表の見出しを使って関係を機械可読にする。
- 装飾画像は支援技術から除外し、意味のある画像には文脈に合う代替テキストを提供する。

### 4. キーボードとフォーカスを実装する

- すべての機能をキーボードで完了できるようにする。
- DOM順序と視覚順序を一致させ、論理的なフォーカス順序を維持する。
- 正の`tabindex`を使用しない。通常はネイティブのフォーカス順序を使用する。
- フォーカス表示を消さず、背景に対して認識できる表示を用意する。
- ダイアログ、メニュー、タブなどでは該当するAPGパターンのキー操作とフォーカス管理を実装する。
- 表示変更後にフォーカスを移す必要がある場合だけ移動し、移動先を予測可能にする。

### 5. 状態変化とエラーを通知する

- 入力エラーを色だけで示さず、問題と修正方法をテキストで示す。
- エラー要約から該当フィールドへ移動できるようにする。
- 非同期の成功、失敗、読み込み完了など、フォーカスを移さず伝える必要がある状態を適切なライブリージョンで通知する。
- `aria-live`や`role="alert"`を過剰使用しない。緊急度と更新頻度に合わせる。
- 無効状態、展開状態、選択状態、現在位置を視覚とアクセシビリティツリーの両方へ反映する。

### 6. 知覚可能な表示を作る

- テキスト、非テキストUI、フォーカス表示に必要なコントラストを確保する。
- 色、位置、形、音だけで情報を伝えない。
- 200%の文字拡大と400%相当のズームでも、主要情報と操作を失わないようにする。
- テキスト間隔の上書き、狭いビューポート、縦横切替、OSの高コントラスト設定で破綻しないようにする。
- 動きや点滅を抑え、可能な場合は`prefers-reduced-motion`を尊重する。
- 動画・音声には用途に応じて字幕、トランスクリプト、音声解説を提供する。

### 7. 検証する

次の順序で検証する。自動検査だけで完了としない。

1. HTMLとアクセシブルネーム計算を確認する。
2. 静的解析と自動アクセシビリティ検査を実行する。
3. キーボードのみで主要フローを完了する。
4. ズーム、リフロー、文字間隔、コントラスト、強制色、動きの軽減を確認する。
5. リスクに応じてスクリーンリーダーで主要フローと動的通知を確認する。
6. 実際のブラウザーでエラー経路を含むユーザーフローを確認する。

詳細な検証方法は[testing.md](./references/testing.md)を読む。

## 実装時の禁止事項

- ネイティブ要素で実現できる操作を、クリック可能な非セマンティック要素で代用しない。
- フォーカス可能な要素からアクセシブルネームを削除しない。
- `outline: none`だけでフォーカス表示を消さない。
- 正の`tabindex`でフォーカス順序を修正しない。
- `aria-hidden="true"`を、フォーカス可能な要素またはその祖先へ付けない。
- ARIA属性を、対応する状態更新やキーボード操作なしで追加しない。
- プレースホルダーだけをフォームのラベルとして使用しない。
- 自動検査の合格をWCAG適合の証明として扱わない。

## 参照資料を選ぶ

- 実装パターン、WCAG 2.2 AAの代表的数値、フォーム、画像、ダイアログ、動的通知を扱う場合は[implementation.md](./references/implementation.md)を読む。
- 自動検査、キーボード、スクリーンリーダー、ズーム、回帰テストを設計する場合は[testing.md](./references/testing.md)を読む。
- 実装完了時またはコードレビュー時は[review-checklist.md](./references/review-checklist.md)を読む。

必要な参照だけを読み、同じ説明を成果物へ機械的に転記しない。

## 結果を報告する

検出した問題ごとに以下を示す。

- 対象と再現条件
- 利用者への影響
- 根拠となる基準またはパターン
- 実装した修正、または推奨修正
- 実施した検証と結果
- 未検証の支援技術、ブラウザー、画面、既知の残存リスク

正式監査を実施していない場合は「WCAG準拠」「完全にアクセシブル」と断定しない。
