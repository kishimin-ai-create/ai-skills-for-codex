# Legacy Code Analysis

## 目的

レガシーコードを変更する前に、変更対象と影響範囲を把握すること。

## 手順

### 1. 現状を理解する

コードを変更する前に、現在のコードを理解する。

コードを十分に理解できない場合は、必要に応じて次の方法を使用する。

- メモを取る。
- スケッチを描く。
- コードに印を付ける。
- 試行リファクタリングを行う。
- 使用していないコードを削除する。
- システムのストーリーを話す。
- CRC を使用する。
- 会話を吟味する。

試行リファクタリングは、コードを理解するために行う。

試行によって得た変更をそのまま採用することを目的としない。

コードを読むことで、次の内容を把握する。

- 主要なクラスやモジュール
- それぞれの責務
- 依存関係
- テストの有無
- 変更頻度が高い箇所
- 複雑な箇所
- コードから読み取れる業務ロジック

コードから設計意図を推測する場合は、推測を事実として扱わない。

確実でないものは、確認が必要なものとして扱う。

### 2. 変更点を洗い出す

実現したい変更によって、どのコードを変更する必要があるかを調査する。

変更する必要があるコードについて、どのメソッドやクラスを対象とするか確認する。

次の内容を調査する。

- 変更対象
- 変更対象から呼び出されるコード
- 変更対象を呼び出すコード
- 変更によって影響を受けるコード
- 変更の影響がどこまで伝播するか

必要に応じて影響スケッチを作成する。

変更が多くの場所へ広がる場合は、変更の影響を絞り込める場所を探す。

### 3. 変更範囲を記録する

変更対象と影響範囲を整理し、以降のテスト作成や変更作業の対象範囲を明確にする。
