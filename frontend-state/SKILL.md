# Frontend State

状態は、寿命、所有者、正規データ、更新頻度、失敗時の扱いで分類する。保存先を先に決めず、外部から観測できる契約とデータフローを固定してから実装する。

## 状態設計ワークフロー

### 1. 状態を分類する

- UI状態: モーダル、選択、フォーカス、表示切替など、局所的で再計算可能な状態。
- 共有アプリ状態: 複数画面が参照し、ユーザー操作で更新する設定・セッション表示など。
- サーバーデータ: APIやDBが正規ソースで、取得・再検証・失敗・キャッシュを伴う状態。
- URL状態: 検索、フィルター、ページ番号、ソートなど、共有・再読み込み可能にする状態。
- フォーム状態: 入力値、dirty、validation、送信中、送信結果。
- オフライン状態: 未送信操作、ローカルキャッシュ、同期キュー、競合情報。

### 2. 所有者と保存先を選ぶ

- 局所UI状態は最も近いコンポーネントのstateへ置く。
- 複数コンポーネントが必要とする場合だけContextや状態ストアへ昇格する。
- サーバーデータをUIストアへ複製せず、Query cacheなどサーバーデータ向けの層で管理する。
- 共有・再現が必要な状態はURL、永続設定は目的に合うストレージ、秘密情報はブラウザ永続領域へ置かない。
- 保存先ごとの寿命、容量、同期、削除、プライバシー、SSR可否を明示する。

### 3. 更新とデータフローを設計する

- 1つの正規ソースから派生値を計算し、同じデータを複数ストアへ二重保持しない。
- 状態遷移、イベント、非同期処理、楽観更新、再試行、キャンセル、競合解決を定義する。
- サーバー更新後のinvalidate、再取得、stale判定、ページング、prefetchを決める。
- ローディング、空、エラー、古いデータ表示、再試行、権限変更をUI契約として扱う。

### 4. オフラインと同期を設計する

- 対象操作、許容する古さ、未送信データ、競合、再送、重複適用、認証期限を列挙する。
- まずクライアント側のアプリケーションコードと既存フレームワーク機能で実現できるか確認する。
- Service WorkerはCacheとProxyなど必要な責務に限定し、状態の正規ソースにしない。
- navigator.onLineだけで判定せず、実際のRequestとNetwork error handlingで状態を確認する。
- 同期失敗、再ログイン、データ削除、ユーザーへの通知、ロールバックを定義する。

### 5. 検証する

- リロード、戻る・進む、複数タブ、SSR/水和、低速回線、再試行、同時更新を確認する。
- 正常、空、エラー、権限、期限切れ、オフライン、復帰、競合をテストする。
- 状態ストアの更新回数、不要な再レンダリング、キャッシュヒット、メモリ・ストレージ容量を測定する。
- データの所有者、保存先、寿命、削除条件、残存リスクを設計記録へ残す。

## 安全性の基準

- APIレスポンスを無制限に永続化せず、個人情報・秘密情報をLocalStorageへ保存しない。
- サーバーデータ、UI状態、フォーム値を一つの巨大ストアへ混在させない。
- 楽観更新は失敗時の復元と競合方針がない状態で導入しない。
- オフラインは「成功した」と誤表示せず、未同期・失敗・期限切れを明示する。
- 状態管理変更に、無関係なUI変更・API仕様変更・データ移行を混在させない。

## 出力契約

状態設計または変更結果と併せて、次を報告する。

1. 状態の分類、所有者、正規ソース、保存先、寿命
2. 更新イベント、非同期処理、キャッシュ、再検証、競合
3. オフライン・復帰・失敗・削除・プライバシー方針
4. テスト、性能、SSR/水和、マルチタブへの影響
5. ロールバック、残存リスク、追加確認事項
