# Frontend State Decision Guide

## 保存先

| 状態           | 第一候補                 | 注意点                           |
| -------------- | ------------------------ | -------------------------------- |
| 局所UI         | Component state          | 最小スコープ、再計算可能         |
| 共有UI         | Contextまたは小さなstore | 更新範囲と再レンダーを測定       |
| サーバーデータ | Query cache              | stale、invalidate、再試行、Abort |
| URL状態        | Query/route params       | 共有・再読み込み・履歴           |
| フォーム       | フォーム管理層           | dirty、検証、送信、復元          |
| 秘密情報       | サーバー側セッション     | ブラウザ永続領域へ置かない       |
| オフライン操作 | 同期キュー               | 冪等性、競合、再送、期限         |

## 比較軸

- 寿命、所有者、正規ソース、容量、プライバシー
- SSR・水和・複数タブ・再読み込みとの互換性
- 更新頻度、失敗時表示、再試行、Abort、競合解決
- キャッシュ鮮度、オフライン許容範囲、削除・失効
- 依存ライブラリ、再レンダー、メモリ、デバッグ容易性

## オフライン

- 読み取りキャッシュと未送信操作を分ける。
- 操作は冪等キーや順序を定義し、再送による二重適用を防ぐ。
- 復帰時は認証期限、競合、サーバー側の最新データを確認する。
- Service Workerを導入する場合は、キャッシュ範囲、更新、失効、ロールバックを明示する。

## 採用を見送る条件

- サーバーデータを複数のストアへ無期限に複製する。
- 便利さだけを理由に個人情報や秘密を永続保存する。
- navigator.onLineだけでオンライン状態を断定する。
- 競合・失敗・復元方針なしに楽観更新やオフライン書き込みを導入する。
