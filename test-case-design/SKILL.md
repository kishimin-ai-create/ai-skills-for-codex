# Test Case Design

テストを「実装の確認」ではなく、観測可能な振る舞い・事後条件・不変条件を表す仕様として設計する。プロジェクト固有のテストランナー、DB、HTTP環境、コマンドを先に検出し、Skill内に固定しない。

## モードを判定する

- outline、テストケース設計、test.todo、テスト仕様という依頼は、テストロジックを書かずにアウトラインだけ作る。
- 実装、todoを埋める、テストを実行するという依頼は、既存アウトラインを保ったままテストを実装する。
- モードが不明なら、まずアウトラインを提示し、実装へ進む前に確認する。

## 共通ワークフロー

1. 対象の責務、入力、出力、エラー、状態遷移、外部副作用、データ不変条件を抽出する。
2. リポジトリのpackage.json、テスト設定、既存テスト、テストユーティリティ、ADR、AGENTS.mdを確認する。
3. Unit、Integration、Contract、E2Eの適切なレベルを選ぶ。詳細は[test-levels.md](./references/test-levels.md)を読む。
4. 正常系、異常系、境界値、空値、重複、権限、タイムアウト、再試行、並行実行、ロールバックを必要性に応じて列挙する。
5. テスト名を「〜の時、〜であること」の形式で書き、何を保証するかを名前だけで理解できるようにする。
6. テスト対象の振る舞いを1ケース1目的で分ける。

## outlineモード

- describeの階層とtest.todoのケース名だけを作成する。
- Arrange、Act、Assert、モック、テストデータ、実装コードは書かない。
- 各グループに、テスト対象の契約を説明する短いコメントを添える。
- 不明な仕様は勝手に補完せず、「仮定」「確認が必要な点」「仮定した場合のケース」を分けて出力する。
- 既存のアウトラインがあれば、階層とケース名を維持し、欠落ケースだけを追加する。

## implementモード

1. outlineのdescribe構造とテスト名を変更せず、test.todoを実装する。
2. AAAを空行で分け、Actは1つの主要な振る舞いに限定する。
3. テストデータは共有前提を適切なfixture・beforeEachへ置き、テスト間の独立性を保つ。
4. DBや外部HTTPはプロジェクトの実環境・テストコンテナ・契約サーバーを優先する。Unitでの小さなfakeやstubまで一律禁止しない。
5. 永続化を伴う場合、戻り値と更新後状態を検証し、異常系では実行前後で状態が変わらないことを確認する。
6. 利用可能なテストランナーのライフサイクルフックで確実にクリーンアップする。onTestFinishedなどのAPIはバージョンと設定を確認してから使う。
7. 実行前に利用可能なスクリプトと対象ファイルを確認し、最小範囲からテストを実行する。
8. 失敗を「テストの誤り」「実装の誤り」「仕様・環境の不明」に分類する。仕様不明のケースは黙ってスキップせず、理由と確認事項を報告する。

## 品質基準

- 実装詳細ではなく公開契約を検証する。
- エラーメッセージだけでなく、エラー種別、HTTPステータス、状態不変条件を検証する。
- 時刻、乱数、並行性、ロケール、タイムゾーンを必要に応じて固定する。
- 過剰なモック、重複fixture、テスト間共有状態、意味の曖昧なテスト名を避ける。
- 実装変更と仕様変更を同じ差分へ混ぜない。

## 出力契約

テスト設計または実装結果と併せて、次を報告する。

1. 対象契約、前提、仕様の不明点
2. テストレベルと選択理由
3. ケース一覧または変更したテストファイル
4. 実行したコマンドと結果
5. 未実装・スキップ・残存リスク・追加確認事項
