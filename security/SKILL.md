# Security

セキュリティに関わる変更では、資産と信頼境界を確認し、認証情報・トークン・秘密情報を漏えいさせない設計を適用する。言語やフレームワーク固有の実装は対象リポジトリの公式設定・ライブラリ規約に従う。

## Workflow

1. 保護対象、攻撃面、信頼境界、データの流れを特定する。
2. 既存の認証・認可、秘密情報管理、入力境界、ログ、依存関係の実装を確認する。
3. [references/security-rules.md](references/security-rules.md)を読み、該当する規則を適用する。
4. 最小変更で実装し、認証情報・トークン・秘密情報が保存、応答、ログ、分析イベントへ出ないことを確認する。
5. セキュリティ関連の正常系、異常系、境界値、認可失敗、情報漏えい防止テストを実行する。
6. 対策、検証結果、残存リスク、未実行の確認を報告する。

## Non-negotiable rules

- パスワードは必ずランダムsalt付きでハッシュ化し、平文を保存・ログ・応答しない。
- パスワード比較はタイミング攻撃に耐える比較を使用し、文字列の直接比較をしない。
- 認証失敗時はアカウント存在を推測できるメッセージや応答差を返さない。
- 秘密鍵、APIキー、DB認証情報、JWT秘密値をリポジトリへコミットしない。
- API境界で入力を検証し、クライアント提供の識別子だけで認可しない。
- パスワード、トークン、認証情報をログへ出力しない。
- セキュリティ対策をUIの非表示やクライアント側ガードだけに依存しない。
