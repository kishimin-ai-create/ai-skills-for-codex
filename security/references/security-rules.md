# セキュリティ規則

## パスワードのハッシュ化

### 必須事項

| 要件                 | 詳細                                                                                                                  |
| -------------------- | --------------------------------------------------------------------------------------------------------------------- |
| 保存前にハッシュ化   | プラットフォームで承認されたscryptまたはbcryptなど、レビュー済みのパスワードハッシュ実装を使用する。                  |
| ランダムsalt         | パスワードごとに一意のsaltを生成する。実装が明示的なsaltを必要とする場合は、最低16バイトを使用する。                  |
| タイミングセーフ比較 | 実装が提供する定時間比較を使用する（Node.jsでは`crypto.timingSafeEqual`）。ハッシュに対して`===`や`!==`を使用しない。 |
| 平文を扱わない       | パスワードを平文で保存、ログ出力、エラーへの含有を行わない。                                                          |
| ハッシュを応答しない | クライアント向けのすべての応答からパスワードハッシュを除外する。                                                      |

### Node.jsの例

```typescript
import { randomBytes, scryptSync, timingSafeEqual } from "crypto";

function hashPassword(plain: string): string {
  const salt = randomBytes(16).toString("hex");
  const hash = scryptSync(plain, salt, 64).toString("hex");
  return `${salt}:${hash}`;
}

function verifyPassword(plain: string, stored: string): boolean {
  const [salt, hash] = stored.split(":");
  const inputHash = scryptSync(plain, salt, 64);
  return timingSafeEqual(inputHash, Buffer.from(hash, "hex"));
}
```

この例は説明用である。実際には対象プラットフォームでレビュー済みの同等実装を使用する。

### 禁止されるパターン

```typescript
// 平文を保存してはならない。
await db.insert(users).values({ passwordHash: input.password });

// パスワードハッシュを文字列比較してはならない。
if (user.passwordHash !== input.password) throw new Error("INVALID");

// パスワードハッシュを返してはならない。
return { id: user.id, email: user.email, passwordHash: user.passwordHash };
```

## 認証エラーの応答

アカウントが存在するかどうかを明らかにしない。存在しないアカウントとパスワード誤りには、同じ処理経路、ステータス、応答形式、同程度の処理時間を使用する。

| 状況                   | 悪いメッセージ    | 必須メッセージ              |
| ---------------------- | ----------------- | --------------------------- |
| アカウントが存在しない | `Email not found` | `Invalid email or password` |
| パスワードが誤っている | `Wrong password`  | `Invalid email or password` |

## トークンと秘密情報

- APIキー、トークン署名用秘密値、データベース認証情報、その他の秘密情報をコミットしない。
- 秘密情報はプラットフォームのSecret管理または環境変数管理へ置き、`.env`、`.env.local`、`.env.production`などの秘密ファイルを`.gitignore`へ追加する。
- 必須変数とプレースホルダーを記載した`.env.example`などの秘匿済み設定例を提供する。
- JWTなどの発行トークンには有効期限を設定する（JWTライブラリでは`expiresIn`を設定する）。有効期限のないトークンを発行しない。
- 長期有効な秘密トークンをブラウザコードへ公開しない。

## 入力検証と認可

- すべての信頼できない入力をAPIまたはプロトコル境界で検証・サニタイズする（ルートハンドラーやバリデーターミドルウェアなど）。
- 必須フィールドの欠落または不正形式をユースケース層へ到達する前に拒否する。
- 所有権と認可をサーバー側で検証し、クライアントが提供した識別子を信頼しない。
- 出力先のコンテキスト（HTML、URL、SQL、シェル、テンプレート、ログ）に応じてエンコードまたはエスケープする。

## パスワード変更

1. 現在のパスワードを要求する。
2. ログインと同じ安全なフローで現在のパスワードを検証する。
3. 検証に失敗した場合は認証失敗（HTTP APIでは通常401）を返して停止する。
4. 新しいsaltで新しいパスワードをハッシュ化する。
5. 保存済みハッシュを置き換える。

現在のパスワードを事前入力または表示してはならない。

## ログ

| 許可                         | 禁止                             |
| ---------------------------- | -------------------------------- |
| エラー種別とステータスコード | 平文パスワード                   |
| リクエストのパスとメソッド   | トークン値                       |
| 不透明なユーザー識別子       | 認証情報を含むリクエスト本文全体 |

## 適用とテスト

コードレビューでは、認証情報の平文保存、ハッシュの直接比較、パスワードハッシュの応答を拒否する。セキュリティロジックには、ハッシュ化・検証、認証情報の存在秘匿、トークン有効期限、入力拒否、認可失敗、機密データのマスキングを対象とする専用テストを用意する。
