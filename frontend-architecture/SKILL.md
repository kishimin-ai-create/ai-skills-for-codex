# Frontend Architecture

フロントエンドの構成を決めるときは、技術名から始めず、外部から観測できる要件と運用制約を固定してから、最も単純な構成を選ぶ。ホスティング、レンダリング、ルーティング、BFF境界を一体の設計として扱う。

## アーキテクチャ選定ワークフロー

### 1. 要件と制約を固定する

- パブリック／プライベート配信、認証、SEO、初期表示、Core Web Vitals、トラフィック変動を記録する。
- データの更新頻度、リアルタイム性、可用性、リージョン、コスト、チームのサーバー運用能力を確認する。
- ルーティング、HTTPS、セキュリティヘッダー、キャッシュ、デプロイ、ロールバックの要件を列挙する。

### 2. BFF/API境界を確定する

- Next.js Route HandlerなどのBFFを使う場合、ブラウザからは同一オリジンの相対`/api/...`を呼ぶ。
- バックエンドのオリジンはサーバー実行時の環境変数から解決し、クライアントへ秘密情報を露出させない。
- 転送先がフロントエンド自身になるプロキシループを拒否する。
- プロキシがレスポンスボディを再送信する場合は、古い`content-length`など転送不能なヘッダーを除去する。
- 直接バックエンドへ接続する設計なら、認証、CORS、タイムアウト、エラー形式を別途明示する。

### 3. レンダリング方式を比較する

- CSR: ログイン後の高度な操作性や開発単純性を優先し、SEOと初期表示を主な要件にしない場合。
- SSR: 一般公開ページでSEOまたは最速の初期表示が重要な場合。Node serverかFaaSかを運用能力で分ける。
- SSG/ISR: ドキュメント、マーケティング、商品詳細など、更新頻度が低く配信性能を優先する場合。
- Hybrid: ルートごとに明確なビジネス価値があり、方式混在の運用・デバッグコストを受け入れられる場合だけ採用する。

### 4. ホスティングと配信を決める

- パブリックな静的配信はCDN＋静的ホスティングを基本にし、キャッシュ無効化とSPAフォールバックを確認する。
- プライベート配信や厳密なヘッダー制御が必要なら、LB＋Web Server、SSR基盤、または同等の制御可能な構成を選ぶ。
- HTTPS、セキュリティヘッダー、認証、ログ、監視、デプロイ／ロールバックが実装できるかを確認する。
- 「フレームワークで簡単」という理由だけでSSRやHybridを選ばない。

### 5. 検証して記録する

- ルートごとのレンダリング方式、データ取得境界、キャッシュ、障害時の挙動を表にする。
- 代表的な端末・ネットワークで初期表示、SEOメタデータ、認証、API転送、セキュリティヘッダーを検証する。
- 選択理由、採用しなかった案、前提、残存リスク、ロールバック手順をADRまたは設計記録に残す。

## 安全性の基準

- ブラウザへ秘密情報、内部オリジン、不要な環境変数を渡さない。
- SPAでHistory APIを使う場合、深いURLの直接アクセスに対するフォールバックを用意する。
- キャッシュによる古いデータ、認証情報の混在、プロキシループ、レスポンスヘッダーの不整合を確認する。
- アーキテクチャ選定と同時に、機能追加・大規模な性能改善・無関係なバグ修正を混ぜない。
