# Frontend Architecture

Webフロントエンドのデプロイモデル、レンダリング戦略、ホスティング構成を選定する際に、このスキルを使用する。

## Core Principles

- パブリックネットワークかプライベートネットワークか、SEO要件、トラフィック特性、運用体制など、ビジネス上の制約から検討を始める。
- ルーティング、セキュリティヘッダー、デプロイ、パフォーマンスの要件を満たせる範囲で、最も単純なアーキテクチャを選ぶ。
- ホスティングとレンダリングは相互に関連する意思決定として扱い、別々に選定しない。

## API Proxy / BFF Boundaries

フロントエンドがNext.jsのRoute Handlerや、その他のBFF形式のプロキシを使用する場合、ブラウザからフロントエンドのオリジンに対して `/api/diaries` のようなリクエストを送る構成は正しい場合がある。

トラフィックが実際にバックエンドへ到達するかどうかは、サーバー側プロキシの転送先を基準に判断する。

- 明示的にバックエンドへ直接アクセスする設計でない限り、ブラウザ側のコードでは相対パスの `/api/...` を使用する。
- バックエンドのオリジンは、サーバー上で実行時環境変数から解決する。
- プロキシループを避けるため、バックエンドのオリジンがフロントエンドのリクエストオリジンと同じ場合は拒否する。
- プロキシランタイムがレスポンスボディを再送信する場合、古い `content-length` や `content-encoding` などのレスポンス形式に関するヘッダーを転送しない。
- 本番相当の動作を確認する際は、バックエンドへ直接アクセスした結果、プロキシ経由の結果、実際に描画されたUIを比較する。

## SPA Hosting Patterns

| Pattern                        | Best for                                     | Strengths                                                                                               | Trade-offs                                                                   |
| ------------------------------ | -------------------------------------------- | ------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------- |
| **CloudFront + S3**            | インターネット公開のSPA                      | エッジ配信、低運用負荷、低コスト、柔軟なレスポンスヘッダー制御、SPAフォールバック対応                   | 閉域・プライベート専用配信には不向き                                         |
| **ALB + S3**                   | プライベートなS3アクセスが必要な限定的ケース | マネージドインフラストラクチャ                                                                          | S3への直接アクセスに伴う負荷、ヘッダー制御の制限、Blue/Green切り替えが難しい |
| **Load Balancer + Web Server** | プライベートまたは閉域ネットワーク向けSPA    | プライベート配信、柔軟なヘッダー制御、SPAフォールバック、ターゲットグループ切り替えによるBlue/Green対応 | サーバー運用が必要                                                           |

### Recommendations

- **インターネット公開のSPA**には、**CloudFront + S3**を選ぶ。
- **プライベートまたは閉域ネットワーク向けSPA**には、**Load Balancer + Web Server**を選ぶ。
- 次の要件がある場合、本番アーキテクチャとしてS3のWebサイトホスティング単体に依存しない。
  - デフォルトでHTTPSが必要
  - カスタムセキュリティヘッダーが必要
  - 実用的なBlue/Greenデプロイが必要

## SSR Hosting Patterns

| Pattern                               | Best for                                                             | Strengths                                                | Trade-offs                                                          |
| ------------------------------------- | -------------------------------------------------------------------- | -------------------------------------------------------- | ------------------------------------------------------------------- |
| **Node server (for example Fargate)** | 安定したトラフィック、十分な運用体制、高いランタイム制御が必要な場合 | 高速なレスポンス、環境差が少ない、柔軟なビルド・デプロイ | 固定費、サーバー運用、スケーリング設計が必要                        |
| **FaaS (for example Lambda)**         | サーバーレス中心のシステム、変動の大きいトラフィック                 | フルマネージド、自動スケーリング、従量課金               | コールドスタート、環境差、静的アセットをCDNやS3へ分離する必要がある |

### Recommendations

- 信頼性、ランタイム制御、運用体制を最優先する場合は、**Node server**を選ぶ。
- トラフィックが急増しやすい、または長時間ほとんどアクセスがなく、サーバーレスによるコスト最適化を重視する場合は、**FaaS**を選ぶ。
- SSRの静的アセットは、すべてをFunctionから配信せず、SSRランタイムと**CloudFront + S3**を組み合わせる。

## Rendering Strategies

| Strategy      | Use when                                                         | Benefits                                                             | Risks                                                         |
| ------------- | ---------------------------------------------------------------- | -------------------------------------------------------------------- | ------------------------------------------------------------- |
| **CSR**       | 高度にインタラクティブなアプリ、ダッシュボード、複雑な社内ツール | 配信構成が単純、クライアント側の操作性が高い、ランタイムコストが低い | 初期表示が遅い、SEOが弱い、クライアント側の攻撃対象領域が広い |
| **SSR**       | SEOと高速な初期表示が重要な場合                                  | 初期表示が速い、クローラーが扱いやすいHTMLを返せる                   | サーバーコストが高い、サーバー側の複雑性が増す                |
| **Hybrid**    | ページごとに異なるレンダリング方式が必要な場合                   | ルートや機能ごとに最適化できる                                       | アーキテクチャとデバッグが複雑になる                          |
| **SSG / ISR** | コンテンツの更新頻度が低い場合                                   | 高いパフォーマンス、低いランタイムコスト                             | 頻繁に変化するデータには不向き                                |

### Recommendations

- SEOとCore Web Vitalsが特に重要な一般ユーザー向けページには、**SSR**を選ぶ。
- 開発の単純さがSEOや初期表示より重要な業務アプリや高度にインタラクティブな画面には、**CSR**を選ぶ。
- **Hybrid rendering**は、追加の保守コストを上回る明確なビジネス価値がある場合にのみ選ぶ。
- ドキュメント、マーケティングページ、商品詳細ページ、その他の更新頻度が低いコンテンツには、**SSG/ISR**を選ぶ。

## Practical Decision Guide

1. **アプリはプライベート専用か**
   - Yes → SPAホスティングには **LB + Web Server** を優先する。
   - No → 次へ進む。

2. **SEOまたは最速の初期表示が必要か**
   - Yes → **SSR**を検討する。
   - No → 次へ進む。

3. **ログイン後の操作が中心の、高度にインタラクティブなアプリか**
   - Yes → **CSR**を優先する。

4. **コンテンツの大半は静的か**
   - Yes → **SSG/ISR**を検討する。

5. **サーバー運用能力があるか**
   - Yes → **Node server SSR**を選択肢にできる。
   - No → **FaaS SSR**を検討する。

## Architecture Anti-Patterns

- フレームワークで簡単に実装できるという理由だけでSSRを選ぶ。
- ルート単位の明確な理由がないままHybrid renderingを使用する。
- History APIを使用するルーティングで、SPAフォールバック要件を無視する。
- プライベート配信やレスポンスヘッダー制御を後付けで考える。
- HTTPSとセキュリティヘッダーが必要な本番環境で、S3のWebサイトホスティングを使用する。
