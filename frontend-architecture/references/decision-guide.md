# Frontend Architecture Decision Guide

## 判断順序

1. 配信対象がプライベート専用なら、LB＋Web Serverなどアクセス制御とヘッダー制御が可能な構成を優先する。
2. 一般公開でSEOまたは最速の初期表示が重要ならSSRを検討する。
3. ログイン後の操作が中心でSEO要件が低ければCSRを優先する。
4. コンテンツが静的または更新頻度が低ければSSG/ISRを検討する。
5. SSRの実行基盤は、Node serverを運用できるか、FaaSの制約を受け入れられるかで選ぶ。

## 比較軸

- SEO、初期表示、インタラクティブ性
- サーバー運用、デプロイ、ロールバック
- キャッシュ無効化、データ鮮度、トラフィック変動
- 認証、CORS、セキュリティヘッダー、内部オリジン秘匿
- コスト、監視、障害時の切り分けやすさ

## 採用を見送る条件

- SSRをフレームワークの既定値だけで採用する。
- ルートごとの明確な理由なしにHybridを導入する。
- History APIのSPAフォールバックを確認しない。
- HTTPS、プライベート配信、ヘッダー制御を後付けにする。
- 要件・制約・運用能力が未確認のままベンダーや製品名を決める。
