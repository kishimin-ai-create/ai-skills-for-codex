# リファクタリングレビュー・チェックリスト

変更の規模とリスクに該当する項目を確認し、未確認の項目は理由を記録する。

## 目的と範囲

- [ ] 目的が構造改善として明確である。
- [ ] 対象範囲と変更しない範囲が記録されている。
- [ ] 新機能、仕様変更、バグ修正、性能最適化を混ぜていない。
- [ ] 現在の公開契約と観測可能な振る舞いを確認している。

## パターンと構造

- [ ] 適用パターンと選択理由が説明されている。
- [ ] 抽出した責務の境界、命名、依存方向が明確である。
- [ ] 抽象化が現在の実利用と変更点に基づいている。
- [ ] Repository、Validator、Interactor、Service、Component、Hookの責務が混在していない。
- [ ] 偶然の類似を過剰にDRY化していない。
- [ ] 処理順と制御フローが意図せず変わっていない。

## 振る舞いと副作用

- [ ] 戻り値、型、ステータス、エラーコード、例外が保持されている。
- [ ] 入力検証、認証、認可、重複確認の順序が保持されている。
- [ ] Repository、外部API、キュー、時刻、乱数、ログの副作用回数が保持されている。
- [ ] タイムアウト、再試行、キャンセル、並行性、リソース解放を確認している。
- [ ] API、Props、設定、シリアライズ、データ形式の互換性を確認している。
- [ ] 秘密情報や個人情報をログ、エラー、テストfixtureへ追加していない。

## テストと検証

- [ ] 必要な特徴づけテストを先に追加した。
- [ ] formatter、lint、型チェックが成功している。
- [ ] 変更対象の単体・統合テストが成功している。
- [ ] 正常、境界、失敗、例外、再試行、権限、並行性を確認している。
- [ ] 関連する利用箇所と回帰テストを確認している。
- [ ] Frontendではアクセシビリティ、フォーカス、ローディング、エラー状態を確認している。
- [ ] 高リスク変更では契約、E2E、負荷、障害、ロールバックを検証している。
- [ ] 性能改善では変更前後を同じ条件で測定している。

## 差分と運用

- [ ] 差分が目的に対して最小である。
- [ ] デッドコード、不要コメント、デバッグ出力、移行途中のフラグが残っていない。
- [ ] ログ、メトリクス、トレースの観測可能性が保たれている。
- [ ] 失敗時のロールバックまたは復旧方法が明確である。
- [ ] 未検証範囲、残存リスク、別作業へ切り出した事項を報告している。
- [ ] コメントはコードから分からないWhy notだけを説明している。
