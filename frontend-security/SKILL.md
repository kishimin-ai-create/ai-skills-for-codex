# Frontend Security

フロントエンドは利用者が制御できる境界であり、機密性・完全性の最終防衛線にしない。脅威、信頼境界、サーバー側の強制点、漏えい時の影響を先に固定してから実装する。

## セキュリティ設計ワークフロー

### 1. 資産と脅威を特定する

- 個人情報、認証情報、決済情報、権限、業務データ、秘密鍵、監査ログを分類する。
- XSS、CSRF、認証回避、権限昇格、IDOR、Open Redirect、クリックジャッキング、依存関係侵害、情報漏えいを検討する。
- ブラウザ、API、BFF、Identity Provider、CDN、ログ、ローカル開発環境の信頼境界を図示する。

### 2. 認証・認可を設計する

- 独自認証より、要件に合うIdentity Providerと標準プロトコルを優先する。
- 認証（本人確認）と認可（操作権限）を分け、認可はサーバーでデータ単位に再検証する。
- セッションCookieはHttpOnly、Secure、適切なSameSite、短い有効期限、ローテーションを検討する。
- ブラウザへ長期秘密トークンを露出させず、OAuth/OIDCのstate、nonce、PKCE、return URLを検証する。
- ログアウト、期限切れ、再認証、複数タブ、パスワード変更後の失効を定義する。

### 3. 入力・出力・ブラウザ保護を実装する

- 入力はサーバー側で型、形式、長さ、範囲、権限、コンテキストを検証する。
- HTML、URL、SQL、シェル、テンプレートへ出力する値はコンテキスト別にエスケープし、危険なHTML挿入を避ける。
- CSRF対策、CORS許可範囲、CSP、HSTS、X-Content-Type-Options、Referrer-Policy、Permissions-Policyを要件に応じて設定する。
- エラー、URL、ログ、分析イベント、ソースマップへ秘密情報・トークン・個人情報を出さない。

### 4. 依存関係と開発環境を管理する

- 依存関係は最小化し、lockfile、脆弱性スキャン、更新方針、サプライチェーンリスクを確認する。
- 開発用プロキシは、外部公開せず、許可された転送先・ヘッダー・Originだけを扱う。
- ローカルのHTTP例外やデバッグ設定を本番へ持ち込まない。環境変数は公開可否を分類する。
- Security Headerを付与できるHostingと、ログ・監視・インシデント対応を選ぶ。

### 5. 検証と対応を記録する

- 認証、権限、セッション、入力、出力、ブラウザ、依存関係、プロキシのテストを用意する。
- 失敗時に機密情報を露出しないこと、認可前にデータへアクセスしないことを検証する。
- 脅威、採用対策、残存リスク、検出方法、ロールバック、インシデント時の失効手順を記録する。

## 安全性の基準

- UIの非表示、クライアントのルートガード、disabled属性だけを認可とみなさない。
- localStorageへ長期アクセストークンを保存する前に、脅威と代替案を明示する。
- CORS、CSP、Cookie、プロキシを「開発で動く」だけで本番安全と判断しない。
- 脆弱性修正と無関係な機能追加・大規模リファクタリングを混在させない。
