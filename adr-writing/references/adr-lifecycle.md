# ADR Lifecycle

## 作成タイミング

次のような設計上の意思決定を行った場合にADRを作成する。

- 責務を変更した
- 依存関係を変更した
- ディレクトリ構成を変更した
- レイヤ構成を変更した
- モジュール境界を変更した
- API設計を変更した
- 共通化・分割を行った
- ライブラリを採用・廃止した
- 今後の実装方針が変わる判断を行った
- AIが同じ間違いを繰り返す可能性がある変更を行った

単なるコード整形やリネームでは作成しない。

人間による設計上の意思決定が確定した後に作成する。

Pull Request とともにレビューを行い、必要に応じて更新する。

## Status

ADRは次のいずれかの状態を持つ。

- Proposed
- Accepted
- Rejected
- Deprecated
- Superseded

リファクタリング後に確定した設計判断を記録する場合は、原則として `Accepted` を使用する。

## ADRの更新

設計判断が変更された場合は、既存のADRを書き換えてはならない。

新しいADRを作成し、既存のADRとの関係を記録する。

既存ADRは、必要に応じて `Superseded` または `Deprecated` に更新する。

誤字、リンク切れなど、意思決定に影響しない修正のみ既存ADRを更新してよい。

## 保存場所

ADRはリポジトリ内の決められたディレクトリへ保存する。

ファイル名には一意の連番と意思決定を表す短い名前を使用する。

例

```text
001-separate-domain-validation.md
002-introduce-anti-corruption-layer.md
```

## 関連付け

ADRには必要に応じて次を関連付ける。

- Pull Request
- Issue
- Commit
- 関連ADR
- 設計資料
- ベンチマーク
- テスト

将来、意思決定の根拠を追跡できる状態を維持する。
