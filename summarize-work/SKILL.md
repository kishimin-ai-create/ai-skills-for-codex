# Summarize Work

作業結果を短くするだけでなく、次の担当者が安全に再開できる監査可能な引き継ぎ情報へ変換する。結論を先に示し、確認できた事実と未確認の仮定を混ぜない。

## 手順

1. 依頼の目的、対象範囲、制約、ADR・AGENTS.md・適用Skillを確認する。
2. 変更前後のファイル、差分、生成物、実行コマンド、テスト結果、エラー、外部操作を収集する。記憶や会話だけで補完しない。
3. 各記述を`FACT`（直接確認）、`ASSUMPTION`（未確認の前提）、`INFERENCE`（事実からの判断）に分類する。未確認情報は断定しない。
4. 完了、部分完了、ブロックを判定する。検証未実行、失敗、ユーザー判断待ちを「完了」に含めない。
5. [テンプレート](references/summary-template.md)に沿って、結論、変更ファイル、検証、事実と仮定、未完了・リスク・次のアクションを記載する。
6. [チェックリスト](references/review-checklist.md)で情報欠失、機密情報、誤った成功表現がないか確認する。

## 記載規則

- 変更ファイルはパスと目的を列挙し、必要なら行番号を示す。
- 検証はコマンドをそのまま示し、pass/fail/not runと失敗理由を記録する。
- 失敗は原因が確定していなければ「原因未確定」と書き、推測を事実として書かない。
- 未コミット、未push、未デプロイは状態を明記する。実行していない操作を実行済みと書かない。
- ログは要点だけを引用し、トークン、Cookie、個人情報、環境変数の秘密値を除外する。
- コードのHow、テストのWhat、コミットメッセージのWhyを混同しない。

## 出力契約

最終出力は、(1)結論、(2)変更、(3)検証、(4)未完了・リスク、(5)次のアクションの順にする。短縮を求められても、失敗・仮定・未検証項目を削除しない。詳細が必要な場合はテンプレートの参照資料へ分離する。
